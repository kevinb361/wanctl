<objective>
Make the Phase 226 matched-load capture harness emit a marked-EF realtime-protection arm
*additively*, without forking it or changing any existing reference-flow invocation. The
candidate (Phase 227) and a fresh besteffort capture both run the identical harness; the only
new behavior is an optional `--marked-ef` arm and the summary fields that read it.

Purpose: AB-04 needs an apples-to-apples realtime-flow comparison (marked EF UDP jitter vs
unmarked UDP, plus unmarked bulk-TCP throughput/latency) under matched load. D-03 forbids a
forked capture script — the matched arms must stay byte-for-byte identical to 226, EF is bolted
on. D-05 makes the degrade-to-best-effort fallback mandatory, not optional. The EF arm is
meaningful because of D-06 (the Phase 225 DSCP-03 verdict: LAN-side EF marks survive to CAKE
ingress on spec-router even though the DOCSIS CMTS strips DSCP downstream — see docs/BRIDGE_QOS.md),
so the arm tests whether diffserv4 acts on that local mark vs besteffort wash ignoring it.

Output: an extended `phase226-baseline-capture.sh` with a `--marked-ef` option, an extended
`phase226-baseline-summary.py` that surfaces the EF arm, and a regression test proving the
matched arms are unchanged.
</objective>

- unmarked TCP:   iperf3 -c "$REF_HOST" -p "$REF_PORT" -t ... --json
                  -> ref-tcp-bulk-unmarked.NN.txt

The default marking mechanic (Claude's Discretion, D-05): iperf3 supports `--dscp ef`
(maps to DSCP 0x2e / TOS 0xb8). If the running iperf3 build rejects `--dscp`, fall back to
`--tos 184`. Record which path was used and whether the egress DSCP was confirmed.
