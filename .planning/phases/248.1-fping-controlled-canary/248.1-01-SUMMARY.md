## Interpretation

Phase 248 was correct that fping is not disqualified by the old Phase 245 `CYCLE_P99_ABS_CEILING_MS=10.0` gate. However, this live native-wanctl canary found a different operational blocker:

**The current native fping integration runs as a 10s background producer, while the 50ms controller loop treats RTT as stale near the producer interval boundary and falls back repeatedly.**

That mismatch causes repeated fallback checks, cycle overruns, and cycle-budget warning alerts. This is a canary failure under the Phase 248.1 gates.

## What must change before another canary

Before retrying a native fping canary, fix or explicitly design around the cadence/staleness mismatch. Plausible paths:

1. Tune fping backend freshness/staleness semantics so a 10s background cadence is not treated as stale by the 50ms loop.
2. Increase fping cadence only if CPU/cycle budget evidence says it is safe.
3. Change the controller's fallback behavior so it does not run expensive fallback checks every 50ms during expected background cadence gaps.
4. Re-run a short canary only after the above is implemented and tested.

Also note that switching from external cake-autorate to native wanctl changes more than RTT backend:

- external cake-autorate Spectrum: 550Mbit download, `rtt 25ms`
- native wanctl Spectrum: 920Mbit download, `rtt 100ms`

A future canary should decide whether that ownership/config delta is acceptable or whether native canary config should be aligned with the external cake-autorate operating point first.
