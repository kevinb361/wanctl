<objective>
Close the two BLOCKED Phase 226 must-haves (AB-02 baseline evidence, GATE-01 threshold lock) without re-capturing any live data. Both failures trace to ONE bug: `parse_tc_qdisc()` only recognizes synthetic summary lines (`Sent N bytes N pkt`, `Dropped|Backlog`, `Avge|Peak delay`) and returns all-zero `TinCounters` against the real CAKE per-tin block present in the retained evidence. That zero summary poisoned `NOISE_BAND_MS.value` (0.0).

Fix the parser to read the real per-tin block, add the regression test that would have caught it, regenerate the baseline artifacts from the intact raw evidence, then recompute and re-provenance the locked noise-band constant.

Purpose: The baseline summary and the GATE-01 tin-separation noise band must be evidentially trustworthy before Phase 227's candidate `diffserv4 wash` deploy. A zero-valued noise band would let any candidate "pass" tin separation trivially.
Output: Corrected parser, real-format regression test, regenerated `baseline-summary.json` + `BASELINE-SUMMARY.md`, and re-provenanced `scripts/phase226-thresholds.json`.

Scope guard: Everything downstream of `parse_tc_qdisc` (`_spread`, `compute_tin_deltas`, delta math, markdown render) already works and MUST NOT be restructured — only row recognition is broken. The cumulative→delta model (`during − before`) is correct because CAKE counters are cumulative since qdisc creation. Deployment is besteffort (single `Tin 0`); `tin_queue_delay_spread_ms` is the spread across the 3 per-run mean `av_delay` values (run-to-run noise band), which the existing `_spread(delays)` call already computes once the parser populates `avg_delay_ms`. Do NOT touch the controller path, CAKE mode, or threshold semantics beyond filling the already-frozen `NOISE_BAND_MS` constant (SAFE-13).
</objective>
