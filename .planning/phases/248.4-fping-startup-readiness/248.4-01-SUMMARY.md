# Plan 248.4-01 Summary — fping Startup First-Sample Readiness

## Outcome

Fixed the remaining native fping startup blocker.

Before this phase, native `wanctl@spectrum` with fping could start the 50ms control loop before fping had published its first cached sample. The control loop treated `measure_rtt() is None` as ICMP failure and immediately ran gateway/TCP fallback checks, producing startup warnings and extra cycle budget work even though fping was simply not ready yet.

Now fping has a bounded first-sample startup grace:

- active backend must be `fping`;
- no cached RTT sample exists yet;
- current time is within `max(5s, cadence_sec * 1.5)` from RTT thread start.

Inside that window, the controller freezes/saves the startup cycle and does not run `handle_icmp_failure()`.

The icmplib path is unchanged.

## Code changes

`src/wanctl/wan_controller.py`:

- tracks `_rtt_thread_started_ts` and `_initial_rtt_pending_logged` when the RTT thread starts;
- adds `_initial_rtt_sample_pending()`;
- changes the no-snapshot path in `measure_rtt()` to log a single INFO readiness message for fping startup instead of repeated WARNs;
- changes `_run_rtt_measurement()` to skip fallback during the fping initial-sample readiness window.

`tests/test_rtt_backend_factory.py`:

- passes `rtt_backend_status` into the test controller so backend-active behavior is observable;
- adds `test_fping_initial_sample_grace_skips_startup_fallback`;
- adds `test_icmplib_initial_none_still_uses_existing_fallback`.
