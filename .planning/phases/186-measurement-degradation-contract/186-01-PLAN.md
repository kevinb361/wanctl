def _build_measurement_section(self, health_data: dict[str, Any]) -> dict[str, Any]:
    """Build current direct-ICMP measurement snapshot for downstream consumers."""
    measurement = health_data.get("measurement", {})
    raw_rtt = measurement.get("raw_rtt_ms")
    staleness = measurement.get("staleness_sec")

        "raw_rtt_ms": round(raw_rtt, 2) if raw_rtt is not None else None,

        "active_reflector_hosts": measurement.get("active_reflector_hosts", []),
        "successful_reflector_hosts": measurement.get("successful_reflector_hosts", []),
    }

## Locked Contract (v1.38 Phase 186)

This section is the version-pinned source of truth for Plans 186-02 and 186-03. It locks the measurement-health payload contract inline in this plan and cross-references Decisions D-01 through D-16 from `.planning/phases/186-measurement-degradation-contract/186-CONTEXT.md`.

### 1. Field List

Per D-01, D-03, D-04, D-08, D-10, D-12, D-15, and D-16, `wan_health[wan].measurement` gains exactly these additive fields:

- `state: str` with legal values exactly `"healthy"`, `"reduced"`, or `"collapsed"`. No other values are legal.
- `successful_count: int` as a non-negative integer derived from `len(successful_reflector_hosts)`; under the current 3-reflector deployment the practical range is `[0, 3]`, but the contract permits any `int >= 0` so future N-reflector deployments do not require a contract revision.
- `stale: bool` as an independent freshness axis.

### 2. State Derivation

Per D-01, D-02, D-04, and D-15, the literal mapping is:

- `successful_count == 3` -> `state = "healthy"`
- `successful_count == 2` -> `state = "reduced"`
- `successful_count <= 1` -> `state = "collapsed"`

The mapping is exhaustive over `range(0, 4)` for the current deployment. A future `successful_count >= 4` requires a contract amendment before a wider reflector deployment lands.

### 3. Staleness Derivation

Per D-05, D-06, D-07, D-13, and D-14, the literal formula is:

```python
if cadence_sec is None or cadence_sec <= 0 or staleness_sec is None:
    stale = True
else:
    stale = staleness_sec > 3 * cadence_sec
```

`age` and `staleness_sec` are the same value: `time.monotonic() - _last_raw_rtt_ts`. The contract keeps the `staleness_sec` name to avoid ambiguity with any future quorum-age timestamp. This reuses the existing fusion boundary pattern at `src/wanctl/health_check.py:475` where freshness is defined as `_age <= _cadence * 3`, so `staleness_sec == 3 * cadence_sec` is not stale and only values strictly greater than that threshold are stale.

No new YAML tunable is introduced. Unknown cadence, non-positive cadence, and missing `staleness_sec` all produce `stale = True`.

### 4. Independence Invariant

Per D-03, `state` and `stale` are orthogonal. There are exactly **6 legal cross-product combinations** because `3 x 2 = 6` for `{healthy, reduced, collapsed} x {fresh, stale}`. The `successful_count` boundary over `{0, 1, 2, 3}` is a separate boundary partition, not a third axis: `0` and `1` both map to `state = "collapsed"`.

Plans 186-02 and 186-03 must use the "6 legal cross-product combinations + boundary partition" framing exactly. No expanded cross-product narrative is part of this contract.

### 5. Backwards-Compatibility Invariants

Per D-10, D-11, and D-12, these existing fields remain verbatim with unchanged semantics:

- `available: bool`
- `raw_rtt_ms: float | None`
- `staleness_sec: float | None` meaning age of the last raw RTT sample, not age of the last successful quorum
- `active_reflector_hosts: list[str]`
- `successful_reflector_hosts: list[str]`

`stale` is additive and does not replace `staleness_sec`. `available` remains the coarser legacy signal and is not deprecated in this phase.

### 6. Section Boundaries

Per D-08 and D-09, only `_build_measurement_section()` at `src/wanctl/health_check.py:383` is augmented by the downstream implementation plan. `_build_reflector_section()` at `src/wanctl/health_check.py:396` remains unchanged, and no new sibling measurement section is introduced.

### 7. Source Of `successful_count`

Per D-01, D-02, D-15, and D-16, `successful_count` is derived in `_build_measurement_section()` as:

```python
len(measurement.get("successful_reflector_hosts") or [])
```

The `or []` coercion is required. If `successful_reflector_hosts` is `None` or absent, the contract coerces it to an empty list, yielding `successful_count = 0` and `state = "collapsed"`. No new controller state or facade method is introduced for this field.

### 8. Source Of `cadence_sec`

Per D-05 and D-13, cadence is sourced controller-side from `self._cycle_interval_ms / 1000.0` and threaded into `health_data["measurement"]["cadence_sec"]` at the existing `get_health_data()` build site in `src/wanctl/wan_controller.py:3392`. `_build_measurement_section()` reads `measurement.get("cadence_sec")` and applies the D-14 fallback rule.

`src/wanctl/rtt_measurement.py` is not modified for this contract. No public `BackgroundRTTThread.cadence_sec` accessor is introduced.

### 9. Requirement Trace

Per MEAS-01, the machine-readable quorum surface is satisfied by `successful_count` plus `state`. Per MEAS-03, degraded measurement health is surfaced by `stale` plus the non-healthy `state` values.
