# Plan 252-02 — RouterOS Route API Boundary, Idempotence, and Failure Tests

<objective>
Add RouterOS default-route read/enable/disable support behind the existing router client/REST boundary, with mocked tests proving idempotence, comment/ID anchoring, and fail-closed API error behavior. Do not use live RouterOS or ad hoc shell/SSH mutation.
</objective>

<must_haves>
<truths>
- Runtime route operations must go through the existing router integration boundary.
- Phase 252 implementation tests are mock-only; no live RouterOS command is executed.
- Route operations must be idempotent and must not churn RouterOS state/logs for already-desired state.
- API failures must not produce false belief that a route state changed.
</truths>
<outputs>
- REST client supports `/ip route print detail where ...` command parsing for route reads.
- REST client supports route enable/disable by exact comment and/or `.id` using tested RouterOS REST endpoints.
- Tests cover success, noop, no match, ambiguous match, and API failure.
- Router client protocol/factory remains coherent if route methods are added.
</outputs>
</must_haves>

<threat_model>
T-252-06 hot-path-shell-mutation: Route operations are implemented by shelling into RouterOS or using the infra-ansible read-only wrapper. Mitigation: add operations to `RouterOSREST`/router client boundary; tests mock `requests.Session` only.

T-252-07 non-idempotent-route-churn: Repeated enable/disable decisions POST every cycle even when route state is already desired. Mitigation: read route state first and return noop success when `disabled` already matches requested state; tests assert no POST occurs.

T-252-08 ambiguous-route-anchor: Comment selector matches multiple routes and mutates the wrong route. Mitigation: zero or multiple matches return failure with no POST; tests cover both.

T-252-09 false-success-on-api-failure: GET/POST failure is reported as changed. Mitigation: methods return non-zero/failed result and include error text; tests assert failure and no false `status: ok`.

T-252-10 SAFE-19-live-mutation: Development accidentally runs live route commands. Mitigation: all verification uses unit tests with mocked session; no live hostnames/secrets/infra wrappers are used.
</threat_model>

<action>
Use RED-GREEN-REFACTOR.

Add failing tests for route mutation command support using mocked REST responses only:

1. `run_cmd('/ip route disable [find comment="Spectrum"]')` GETs `/rest/ip/route`, finds exactly one enabled route with comment `Spectrum`, then POSTs to `/rest/ip/route/disable` with `{ ".id": route_id }` and returns success metadata.
2. `run_cmd('/ip route enable [find comment="ATT"]')` finds exactly one disabled route and POSTs to `/rest/ip/route/enable`.
3. Disabling a route already disabled returns success/noop metadata and makes no POST.
4. Enabling a route already enabled returns success/noop metadata and makes no POST.
5. Zero matches returns command failure and makes no POST.
6. Multiple matches returns command failure and makes no POST.
7. GET failure returns command failure and makes no POST.
8. POST failure returns command failure and does not report changed/success.
9. If `.id` anchoring is added, `run_cmd('/ip route disable *6')` or a documented supported command form is tested; otherwise explicitly leave ID anchoring as a Phase 253/254 config-use detail and document comment anchoring for Phase 252.

Implement in `src/wanctl/routeros_rest.py`:

- Extend `_execute_single_command()` to route `/ip route enable`, `/ip/route/enable`, `/ip route disable`, and `/ip/route/disable` to a new route action handler.
- Add route lookup helper using exact comment or `.id`, with failure on zero/multiple matches.
- Determine current disabled state from RouterOS response values (`"true"`, `"false"`, bool, missing) conservatively.
- For desired state, return noop success if route already has desired disabled value.
- Use POST command endpoints `/ip/route/enable` and `/ip/route/disable` with `{ ".id": route_id }` for actual mocked mutation.
- Return structured metadata including route anchor, route id, action, and `changed`/`noop` state.
- Keep `run_cmd()` semantics: if handler returns `None`, `run_cmd()` returns non-zero.
</action>
