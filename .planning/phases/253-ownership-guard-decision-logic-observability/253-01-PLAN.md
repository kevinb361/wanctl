# Plan 253-01 — Netwatch Ownership Guard + Migration Acknowledgement Fail-Closed Tests

<objective>
Add a mock-testable route ownership guard that detects route-mutating Netwatch/script conflicts and keeps active wanctl route mutation fail-closed unless an explicit migration acknowledgement and clean guard result allow it.
</objective>

<must_haves>
<truths>
- Netwatch remains the interim WAN route owner until Phase 254 explicitly accepts a migration/canary result.
- Exactly one component may mutate WAN default routes at a time.
- Active wanctl route mutation must refuse to run while route-mutating Netwatch entries/scripts are enabled unless a deliberately named migration acknowledgement is configured.
- RouterOS guard read failures are unsafe and must block active mutation.
- Phase 253 tests must be mock-only; no live RouterOS or Netwatch command is run.
</truths>
<outputs>
- A guard module returns structured ownership status and conflict evidence.
- Config validation has an explicit active-mode acknowledgement field with fail-closed defaults.
- Example/docs show the acknowledgement as dangerous and disabled/commented, not as the default path.
</outputs>
</must_haves>

<threat_model>
T-253-01 missed-netwatch-conflict: wanctl begins active route mutation while RouterOS Netwatch/scripts still mutate the same routes. Mitigation: guard inspects mocked Netwatch/script data and fails closed on route-mutating enabled conflicts.

T-253-02 unsafe-acknowledgement: a vague or default-on flag bypasses the guard. Mitigation: add an explicit migration acknowledgement key that defaults false, is documented as canary-only, and is required for active mode validation.

T-253-03 guard-read-failure-open: RouterOS read errors are treated as no conflicts. Mitigation: guard result includes `status="error"`, `active_allowed=False`, and failure reason; tests assert active remains blocked.

T-253-04 hard-coded-only-detection: guard only checks current Snapshot-A names and misses other route-mutating scripts. Mitigation: detection scans script source/commands for route enable/disable patterns, while Snapshot-A names remain examples/evidence.

T-253-05 SAFE-19-live-mutation: implementation accidentally runs live RouterOS inspection/mutation. Mitigation: only unit tests with fake router clients are used; no production config, systemd, Netwatch, or RouterOS state is changed.
</threat_model>

Test concrete behavior:

1. `RouteOwnershipGuard.inspect()` returns `status="ok"`, `active_allowed=True`, `conflicts=[]` when mocked Netwatch entries are disabled or scripts do not contain route mutation commands.
2. It returns `status="conflict"`, `active_allowed=False`, and at least one conflict when a mocked enabled Netwatch entry references a script whose source contains `/ip route disable` or `/ip route enable`.

Use the existing router client `run_cmd()` boundary with mocked command strings only. Keep command strings read-only in intent, for example `/tool netwatch print detail` and `/system script print detail`; do not add enable/disable commands in the guard.
