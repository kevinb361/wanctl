**Step 1: Pre-flight config diff (per D-discretion, feedback_diff_config_before_deploy.md)**

```bash
echo "=== Spectrum config diff ==="

echo "=== ATT config diff ==="
diff <(ssh kevin@10.10.110.223 "sudo cat /etc/wanctl/att.yaml") configs/att.yaml
echo "Exit code: $?"
```

Accept/reject criteria for diffs (addresses MEDIUM review concern):
- **Accept (proceed):** No diff (exit 0), or diff is only whitespace/comments
- **Accept with note:** Diff is a known production tuning override (e.g., tuning_params differ from YAML defaults -- these live in metrics.db, not YAML)
- **REJECT (stop):** Diff shows changed thresholds, endpoints, IP addresses, or db_path values that would break the deploy. Resolve the diff before continuing.

**Step 2: Check storage pressure BEFORE migration (addresses MEDIUM review concern)**

```bash
ssh kevin@10.10.110.223 "curl -sf http://127.0.0.1:9101/health | python3 -c \"import sys,json; d=json.load(sys.stdin); print('pre-migration storage:', d.get('storage',{}).get('status','MISSING'))\""
```

Note the pre-migration storage status for comparison after migration. If status is already `critical`, the migration is even more important -- proceed.

**Step 5: Storage migration if needed (per D-05, D-06)**

```bash
ssh kevin@10.10.110.223 "sudo test -f /var/lib/wanctl/metrics.db.pre-v135-archive && echo 'ALREADY_MIGRATED' || echo 'NEEDS_MIGRATION'"
```

```bash

```

Note: migrate-storage.sh stops both wanctl services at line 114 (`systemctl stop wanctl@spectrum.service wanctl@att.service || true`). Since Spectrum is already stopped, this is a no-op for Spectrum. ATT may be briefly stopped by the migration script -- this is acceptable because the migration archives the shared metrics.db that both services previously used. The migration is inherently global (single shared DB). ATT will be restarted in Plan 03.

If ALREADY_MIGRATED: skip (per D-06).

**Step 6: Start Spectrum service**

```bash
ssh kevin@10.10.110.223 "sudo systemctl start wanctl@spectrum.service"
```

**Step 7: Wait for Spectrum health readiness (addresses MEDIUM fixed-sleep concern)**

Use health endpoint readiness instead of fixed sleep. Poll until the endpoint responds with valid JSON and uptime >= 10s:

```bash
for i in $(seq 1 15); do
  HEALTH=$(ssh kevin@10.10.110.223 "curl -sf http://127.0.0.1:9101/health" 2>/dev/null) && \
  UPTIME=$(echo "$HEALTH" | python3 -c "import sys,json; print(json.load(sys.stdin).get('uptime_seconds',0))" 2>/dev/null) && \
  if [ "$(echo "$UPTIME >= 10" | bc -l 2>/dev/null || echo 0)" = "1" ]; then
    echo "Spectrum ready: uptime=${UPTIME}s"
    break
  fi
  echo "Waiting for Spectrum readiness... (attempt $i, uptime=${UPTIME:-unknown}s)"
  sleep 2
done
```

**Step 8: Validate Spectrum health (per-WAN canary)**

```bash

```

Expected output:
- version: 1.35.0
- storage: ok (fresh per-WAN DB after migration should report ok; if warning, investigate DB size before proceeding)
- download: GREEN or YELLOW (not unknown)

The second mtime should be greater than the first, confirming active writes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dev machine -> cake-shaper VM | SSH with key auth, sudo on remote |
| deploy.sh rsync -> /opt/wanctl | Code deployment overwrites production binaries |
| health endpoint -> validation | Trusts health JSON from localhost; version is a release gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-173-02 | T (Tampering) | deploy.sh rsync | accept | SSH key auth + rsync checksums; deploy to known host only |
| T-173-03 | D (Denial of Service) | Spectrum service stop | mitigate | Only Spectrum stopped, ATT remains active providing WAN redundancy during deploy window |
| T-173-04 | I (Information Disclosure) | config diff output | accept | Config diff runs on dev machine, no secrets in YAML (secrets in /etc/wanctl/secrets) |
| T-173-05 | T (Tampering) | version mismatch | mitigate | Health endpoint version checked immediately after start; mismatch triggers rollback before ATT is touched |
| T-173-06 | D (Denial of Service) | migration stops ATT | accept | migrate-storage.sh may briefly stop ATT (line 114) as side effect of archiving shared DB; ATT restart handled in Plan 03 |
</threat_model>
