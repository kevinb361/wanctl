**Step 1: Run canary (per D-07)**

```bash
./scripts/canary-check.sh --ssh kevin@10.10.110.223 --expect-version 1.35.0
echo "CANARY EXIT CODE: $?"
```

Exit code MUST be exactly 0. This is the DEPL-01 acceptance gate.

Exit code interpretation:
- **0 (PASS):** ERRORS=0 AND WARNINGS=0. DEPL-01 satisfied.
- **1 (FAIL):** Endpoint unreachable, router unreachable, storage critical, runtime critical. BLOCKING. Investigate immediately.
- **2 (WARN):** Version mismatch, uptime < 10s, storage warning, runtime warning, SOFT_RED/RED state. Does NOT satisfy DEPL-01.

**If exit code is 2 (WARN):**

Check the canary output for which WAN and which check triggered WARN:

- `storage_status == "warning"`: A fresh per-WAN DB after migration should read ok. If warning, investigate DB size (`sudo ls -lh /var/lib/wanctl/metrics-*.db`). This is a BLOCKING issue -- do NOT declare DEPL-01 satisfied.
- `uptime < 10s`: Wait longer, rerun canary. This is transient.
- `version mismatch`: Investigate which WAN reports wrong version. This is a BLOCKING issue.
- `download/upload SOFT_RED or RED`: This is transient network congestion. If canary exits 2 SOLELY due to congestion state (all other checks pass), wait a few minutes for congestion to clear and rerun. Document this as expected possibility per Pitfall 8.
