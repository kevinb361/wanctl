<objective>
Build the factory module — the only genuinely new code in the phase. It selects
icmplib vs fping from measurement.backend, falls back to icmplib LOUDLY at
construction time when fping is selected but absent, and returns an RttBackendHandle
(NOT a raw thread). The handle exposes a DEFERRED make_thread() builder that the
autorate caller invokes later (when the pool + hosts_fn exist) and whose output
satisfies the COMMON THREAD PROTOCOL WANController.measure_rtt() requires. For the
fping path, make_thread builds an in-module ADAPTER wrapping FpingThread so a raw
FpingThread is never assigned to _rtt_thread.

CYCLE-3 HIGH FIX: build_rtt_backend resolves the INDEPENDENT fping cadence from
measurement.fping.cadence_sec (Phase 241 D-06; default 10.0) and stores it on the
handle. The fping branch of make_thread builds FpingThread with THAT resolved fping
cadence — NEVER the caller-supplied controller background-RTT cadence (≈0.25s on the
50ms loop), which would force the timeout-vs-cadence fallback every startup and leave
fping permanently dead. The icmplib branch still uses the caller-supplied controller
cadence as today.

The handle ALSO carries a separate controller_measurement field that is ALWAYS an
icmplib RTTMeasurement — this is the object WANController.rtt_measurement binds to so
the controller's helper paths (ping_host gateway check, ping_hosts_with_results
blocking fallback, maybe_probe reflector scorer — none of which exist on
FpingMeasurement) keep full RTTMeasurement semantics even when the background RTT
thread runs fping. This plan makes the Plan 01 factory unit tests — including the
thread-protocol, the measure_rtt-no-AttributeError, the controller_measurement-is-
RTTMeasurement, the fping-active-with-real-cadence, and the scorer-non-use
regressions — GREEN.

Purpose: Centralize backend construction in one place with zero call-site branching
(D-04), make fallback observable per-WAN (FALL-02), and CLOSE the review findings:
(1) the factory hands the call sites a protocol-satisfying thread handle/adapter, not
a raw FpingThread that would AttributeError in measure_rtt (cycle-1 HIGH, already
closed); (2) the factory hands a guaranteed-icmplib controller_measurement so
WANController.rtt_measurement never becomes a FpingMeasurement (cycle-2 HIGH, already
closed); (3) the fping branch uses the resolved fping cadence so fping actually RUNS
in the live path (cycle-3 HIGH); and (4) fping is constructed without a shared scorer
so it never double-feeds the reflector scorer (cycle-3 MEDIUM).
Output: src/wanctl/rtt_backend_factory.py with build_rtt_backend(), RttBackendHandle
(with controller_measurement + fping_cadence_sec), the fping adapter, the deferred
make_thread builder, and the WARN-once + per-WAN counter helpers; no call-site edits
in this plan.

This plan creates a NEW module only — it does not edit any existing src/wanctl file,
so it does NOT touch the controller path beyond adding one allowlisted file. The
fping adapter lives HERE (inside rtt_backend_factory.py, already allowlisted), NOT
in the 241-frozen fping_measurement.py.
</objective>
