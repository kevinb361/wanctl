<objective>
Capture the actual DSCP distribution arriving at the Spectrum CAKE enqueue interfaces under both
representative organic traffic and a deliberately EF-marked probe flow, satisfying DSCP-02. Because
production CAKE runs `besteffort` (single tin), per-tin CAKE counters cannot answer survival — so
observe the DSCP byte directly at the egress interface via bounded `tcpdump`, histogrammed offline,
read-only. Pair the organic distribution with a known EF probe to prove whether a specific mark
survives to the shaper.
</objective>

<context>
DL CAKE egress is `spec-router`, UL CAKE egress is `spec-modem`, both on host `cake-shaper`
(10.10.110.223). The bridge `spectrum_dl` chain marks DL flows just upstream of `spec-router`; the
question is whether those marks (and endpoint EF marks) reach the enqueue point intact.

**Cycle-1 + cycle-2 HIGH corrections shape this plan:**
1. **Capture point vs CAKE wash (cycle-1 VERIFIED) + capture-point PROOF (cycle-2 HIGH-C VERIFIED).**
   Production runs `diffserv: besteffort` + `allow_wash: true` (configs/spectrum.yaml:44-45). Wash
   strips DSCP AFTER tin selection at enqueue. A capture taken on the post-wash transmit side would
   falsely show EF as stripped even though it reached CAKE intact — a false negative that closes the
   milestone. So the capture MUST observe the DSCP byte as it ARRIVES at the enqueue interface
   (pre-wash). CRITICAL (cycle-2 HIGH-C): the naive `-Q in` recipe is directionally WRONG for DL —
   DL packets EGRESS `spec-router` toward the LAN, so host-inbound on `spec-router` is NOT the
   pre-CAKE enqueue point. The correct DL point is inbound on the `spec-router` bridge member facing
   `spec-modem`/the `spectrum_dl` marking chain, derived from live topology, NOT assumed.
   `CAPTURE_POINT` is a PROVEN property backed by `capture-point-proof.txt`, defaulting to `unknown`
   until the proof substantiates `pre_wash_ingress`.
2. **Direction split + topology under test (HIGH).** A UL probe sourced on cake-shaper only proves a
   local upload mark reaches `spec-modem`; it does NOT exercise the CRS/Ruckus/bridge DL
   classification path the milestone re-evaluates. So evidence is split into four channels —
   `organic_dl_histogram`, `organic_ul_histogram`, `dl_ef_probe`, `ul_ef_probe` — and only the DL
   channels can drive the DL diffserv verdict.

`tcpdump` reading the IP TOS/DSCP byte (handling Ethernet/VLAN/IPv6/cooked offsets) gives a
read-only per-DSCP histogram without touching the qdisc filter list. EF survival is decided on the
probe's own 5-tuple (a probe-only filtered pcap), by count/percentage — not by unfiltered histogram
presence (background EF or one stray packet must not flip the result). The probe needs an
operator-supplied `--probe-target`; it must never default to an implicit/arbitrary host. The
read-only constraint forbids gear-config mutation and CAKE-mode change — it does not forbid benign
bounded diagnostic traffic, consistent with prior flent/RRUL capture phases.
</context>

<title>Write the read-only DSCP-at-ingress histogram capture script</title>
<action>
Create `scripts/phase225-dscp-ingress-capture.sh` (`set -euo pipefail`) with args `--output-dir`
(required), `--ssh-host` (default `cake-shaper`), `--duration` (default 90 seconds),
`--probe {none|ul|dl|both}` (default `none` — a probe is opt-in and requires a target),
`--probe-target <host>` (REQUIRED whenever `--probe` is not `none`; the script errors with a clear
message if a probe is requested without a target — it must never default to an implicit/arbitrary
host), `--probe-proto {udp|tcp}` and `--probe-port` (recorded into the 5-tuple), and a
`--min-packets` (default 2000) / `--min-active-seconds` (default 30) sample floor.

**Capture-point proof FIRST (HIGH-C — prove, do not assert):**
- Before any histogram capture, derive the correct DL pre-wash observation point from the LIVE
  topology, not from an assumed `-Q in` recipe. DL packets EGRESS `spec-router` toward the LAN, so
  host-inbound (`-Q in`) on `spec-router` is NOT the pre-CAKE enqueue point. The correct DL point is
  inbound on the `spec-router` bridge member that faces `spec-modem`/the `spectrum_dl` marking chain
  (the byte after the bridge sets DSCP but before the `spec-router` egress qdisc washes it). Read
  `ip -d link show`, the bridge member roles, and the `iif spec-modem oif spec-router` classification
  path to identify the interface + capture direction.
- Emit `capture-point-proof.txt` per direction citing the topology evidence used (interface name,
  capture direction, bridge member role, and the qdisc location relative to the capture hook) that
  justifies the chosen point. Only when this proof substantiates that the hook sits downstream of the
  DSCP-setting stage AND upstream of the CAKE egress/wash qdisc may the script record
  `CAPTURE_POINT=pre_wash_ingress` + `WASH_ORDERING_PROVEN=true` for that direction. **Default until
  proven is `CAPTURE_POINT=unknown` / `WASH_ORDERING_PROVEN=false`** → that channel is `unknown` for
  the verdict and cannot drive MARKS_DO_NOT_SURVIVE.

**Representative (organic) capture, direction-split + pre-wash:**
- DL on the proven `spec-router` pre-wash interface/direction and UL on `spec-modem`, each via
  `ssh -o BatchMode=yes "$SSH_HOST" "sudo -n timeout -k 5 <duration> tcpdump -n -p -i <proven_if> <proven_direction_flag> -c <cap> -w -"`
  capturing the pre-wash ingress-to-enqueue direction established by the proof step above (the
  direction flag is whatever the proof determined — NOT hardcoded `-Q in`). Save raw pcaps under
  `--output-dir/raw/` and compute direction-tagged per-DSCP histograms
  `dscp-histogram-spec-router-dl.txt` and `dscp-histogram-spec-modem-ul.txt` via a python3 extractor
  that correctly derives DSCP from the IP header across Ethernet, 802.1Q VLAN tags, IPv6 Traffic
  Class, and Linux cooked (SLL) captures — NOT a naive `ip[1] >> 2` that breaks on VLAN/IPv6/cooked
  frames. Each histogram MUST emit BOTH `NONBE_PKT_PCT=` (non-BestEffort packet fraction) AND
  `NONBE_BYTE_PCT=` (non-BestEffort byte fraction, where bytes are measurable) so 225-03 can apply
  the full pre-registered NEGLIGIBLE criterion (packets < 1.0% AND bytes < 1.0%), not packets alone.
- `timeout` exits 124 when the duration elapses before the packet cap — handle this explicitly
  (treat 124 as a normal end-of-window, not a failure) so the script does not abort under `set -e`.
- Compute and record sample-quality metrics per direction into `sample-quality.txt`: total IP
  packets, total IP bytes, active seconds, `NONBE_PKT_PCT`, `NONBE_BYTE_PCT`, and
  `WINDOW_VALID_DL=true|false` / `WINDOW_VALID_UL=true|false` against the `--min-packets` /
  `--min-active-seconds` floor. A window below either floor is recorded invalid (quiet-window guard)
  and the verdict treats it as `invalid`, never as evidence of absence.

**EF probe, direction-split, 5-tuple decided:**
- `--probe ul`: low-rate EF flow from cake-shaper outbound to `--probe-target`
  (`iperf3 -u -b 1M --tos 0xb8 -c <target>` or `nping --udp --tos 0xb8 --rate 50 <target>`),
  captured pre-wash on `spec-modem`. This is `ul_ef_probe`.
- `--probe dl`: EF flow on the DL/return path that traverses the CRS/Ruckus/bridge classification
  path under test (probe target/return toward a LAN endpoint), captured pre-wash on `spec-router`.
  This is `dl_ef_probe` — the channel the milestone actually re-evaluates.
- For each requested direction, write a probe-only pcap filtered to the probe 5-tuple under
  `raw/`, then decide survival by COUNT/PERCENTAGE on that filtered pcap: emit per-direction
  `dl-ef-probe-result.txt` / `ul-ef-probe-result.txt` with `PROBE_5TUPLE=...`, `PROBE_PKTS_SENT=N`,
  `PROBE_PKTS_CAPTURED=M`, `EF_PKTS_AT_ENQUEUE=K`, `EF_SURVIVAL_PCT=<pct>`, and
  `EF_SURVIVED=true|false|degraded` per the pre-registered thresholds (≥90% → true, <10% → false,
  else degraded; and `unknown`/degraded if fewer than 100 probe packets were emitted+captured). If
  the rig cannot mark EF cleanly (DL WMM strip), set `degraded` and record the observed DSCP.

Write MANIFEST.md and a required-artifact assertion loop. The script MUST use `tcpdump` only (no
`tc filter add`/`tc qdisc` mutation), MUST NOT change CAKE mode, and MUST NOT reference
controller-path source files.
