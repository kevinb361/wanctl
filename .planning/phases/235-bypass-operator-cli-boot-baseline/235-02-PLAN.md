Baseline 5-verb sequence + read-back substrings (docs/SILICOM-BYPASS.md lines 540-578, authoritative for THIS card/firmware). Encode the EXACT documented strings — several are counterintuitive double-negatives:
  set_dis_bypass off   -> get_dis_bypass    expect "Bypass mode enabled"      (clears the *disable* bit — NOT "disabled")
  set_bypass_pwoff on  -> get_bypass_pwoff  expect "Bypass at power off"
  set_bypass_pwup off  -> get_bypass_pwup   expect "non-Bypass at power up"
  set_disc_pwup off    -> get_disc_pwup     expect "non-Disconnect at power up"
  set_std_nic off      -> get_std_nic       expect "not in Standard NIC mode"
Applies to BOTH pairs from PAIRS ("att-modem spec-modem").

MANDATORY per-pair readiness poll (Review fix: 235-REVIEWS 235-02 HIGH #3). `After=bpctl-silicom.service`
proves module/device readiness, NOT that att-modem/spec-modem are ready for per-iface bpctl calls. So the
baseline MUST, before applying to each pair, run a BOUNDED retry that probes the pair is bpctl-capable
(e.g. `$BPCTL_UTIL <pair> get_bypass_slave` returns non-empty, or `ip link show <pair>` succeeds) — the
poll is NON-OPTIONAL. If a configured pair never becomes capable within the bound, baseline fails loudly
(non-zero exit + journal error) rather than skipping or proceeding.

Read-before-set (Review fix: 235-REVIEWS 235-02 MEDIUM — reduce repeated card writes / EEPROM/NVRAM wear).
For each of the 5 verbs, read the current get_* FIRST; if it already matches the documented want string,
SKIP the set_* (do not write) and just confirm; otherwise set_*, then re-read and assert. This still
satisfies BOOT-01 (every setting is read-back-asserted) while avoiding unnecessary card writes every boot.

Existing boot unit to reconcile (deploy/systemd/bpctl-silicom.service — full file):
  [Unit] After=local-fs.target systemd-udev-settle.service ; Wants=systemd-udev-settle.service ; Before=wanctl@att.service wanctl@spectrum.service
  [Service] Type=oneshot ; RemainAfterExit=yes ; ExecStart=/usr/local/sbin/wanctl-bpctl-init
  [Install] WantedBy=multi-user.target
  Owns: bpctl module load + /dev/bpctl0. KEEP this responsibility; do NOT rebuild wanctl-bpctl-init.

New init unit target ordering (RESEARCH Pattern 3 / Assumption A3):
  Requires=bpctl-silicom.service ; After=bpctl-silicom.service
  Before=cake-autorate-att.service cake-autorate-spectrum.service wanctl@att.service wanctl@spectrum.service
  Type=oneshot ; RemainAfterExit=yes ; ExecStart=/usr/local/sbin/silicom-bypass baseline
  WantedBy=multi-user.target
  Do NOT reference systemd-udev-settle (deprecated). Live mode is cake-autorate; wanctl@ units are disabled but harmless to list.
  ORDERING-ONLY model (Review fix: 235-REVIEWS 235-02 MEDIUM — make the relationship explicit). Before=/After=
  express ORDERING, not activation: they only take effect when the listed units are in the same boot
  transaction. The init unit is pulled into boot by its own WantedBy=multi-user.target (it must be ENABLED on
  the host, which is the Plan 03 operator step), NOT by a Wants= from the WAN units. Document this in a unit
  comment so an operator knows the init unit must be enabled alongside the WAN services, not merely present.

Bounded master-iface poll model (scripts/wanctl-bpctl-init:18-24): 50×0.1s loop checking existence before acting.
