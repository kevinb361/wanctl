<objective>
Prove ACCESS-02 against the live router: with the new REST netwatch handler in place (Plan 02), a
live `/ip route print` AND `/tool netwatch print` each return exit 0 with non-empty, parseable output
from `cake-shaper` (D4 — single-command proof rejected). Enforce ACCESS-03/SAFE-21 by validating
every live command through the generalized read-only validator first.

Critical prerequisite (the A1 gate): RESEARCH assumption A1 is unverified — does the deployed
RouterOS at 10.10.99.1 actually expose `/rest/tool/netwatch`? This plan opens with an operator
read-only check. If it returns parseable JSON, the REST-handler approach is confirmed and the proof
proceeds. If it 404s, the plan forks to the SSH `router.key` fallback (Pattern 2) and records that
fork instead of falsely declaring ACCESS-02 proven over REST.

Purpose: D4 is the ACCESS-02 acceptance gate and cannot be satisfied by unit tests alone — it needs
a live two-read proof. Proving both reads now de-risks Phase 259, which needs exactly these two reads.

Output: a validated read-only command file and the D4 proof evidence (or the A1-fail fork record).

NOTE: This plan is operator-driven. Privileged reads on `cake-shaper` are handed to Kevin as
`! <command>` per D3 — never via credential escalation. SAFE-21: read-only `print`/GET only; no
RouterOS mutation, no Netwatch change, no steering.service restart.
</objective>
