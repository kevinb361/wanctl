<objective>
Extend `scripts/phase226-baseline-summary.py` with an additive `tin_separation` block so the
GATE-01 tin-separation gate (D-05 occupancy + D-06 delay-gap) can be read from the diffable
summary rather than re-parsed from tc inside the verdict evaluator (D-02). Then re-emit BOTH
authoritative summaries in place from their already-captured tc artifacts — read-only re-run,
no recapture, no production touch.

Purpose: the gate-eval (plan 228-02) consumes summary JSON, not raw tc. Today the summary's
`interfaces` block carries per-tin queue-delay/backlog/packets, but it does NOT compute the
non-BE-vs-BE separation metric the threshold set evaluates (D-05 `non_be_packets`/`non_be_sojourn`
occupancy, D-06 `per_tin_queue_delay`/`per_tin_backlog` delay gap vs `NOISE_BAND_MS`). This plan
builds that derived metric so gate-eval input stays uniform (summary-in -> verdict-out) and the
separation signal is diffable. The diffserv4 candidate summary has tins 0..3; the besteffort
baseline collapses to tin 0 only — "no separation" is the correct besteffort result, not an error.

Output: an extended summary generator emitting `tin_separation`, both authoritative summaries
re-emitted with that block, and a regression test locking the multi-tin and single-tin behavior
plus pre-existing-field preservation.
</objective>

The summary generator today (`scripts/phase226-baseline-summary.py`):
- `parse_tc_qdisc()` -> `TinCounters` per tin (packets, drops, backlog_bytes, delay).
- `compute_tin_deltas(before, during, after)` -> per-tin before/during/after deltas + a
  `tin_queue_delay_spread_ms` per tin (computed for the MD table).
- The summary JSON `interfaces` block is `{iface: {tin_id: {mean_*_delta, runs[], tin_queue_delay_spread_ms, stddev_ms}}}`.
- It ALSO emits `baseline_window` (restart_rate, transition_rate, floor_hit_cycles, soft_red_dwell_s),
  `rrul_p99_latency_under_load_ms_mean`, `marked_ef`, `ref_udp_unmarked`, `ref_tcp_unmarked`,
  `provenance`, `schema_version`, `run_count`.

SOURCE FACT (verified): the besteffort authoritative summary
(`baseline-20260604T154929Z/baseline-summary.json`) has only tin "0" under each interface —
besteffort wash collapses all traffic into BestEffort. The candidate (`candidate-20260604T163152Z`)
is diffserv4 and is expected to carry tins 0..3. There is no derived non-BE-vs-BE separation
metric in the JSON today; that is what this plan BUILDS.

`scripts/phase226-thresholds.json` -> `TIN_SEPARATION`:
- OCCUPANCY_REQUIRED + OCCUPANCY_METRICS [non_be_packets, non_be_sojourn]
- DELAY_GAP_REQUIRED + DELAY_GAP_METRICS [per_tin_queue_delay, per_tin_backlog]
- MAGNITUDE_RULE: "non-BE vs BE per-tin delay gap must exceed the Snapshot A baseline 3-run spread"
- NOISE_BAND_MS.value = 24.206 (read this from the JSON; do NOT hardcode the literal in the .py)

CAKE tin convention: tin "0" is BestEffort. Tins 1..N are the non-BE diffserv tins. "Non-BE
occupancy" = any non-zero packet/sojourn presence in tins != "0". The D-06 delay gap = (max non-BE
mean per-tin delay) - (BE tin mean per-tin delay); useful separation requires that gap to exceed
NOISE_BAND_MS AND non-BE occupancy to be present.
