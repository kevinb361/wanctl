<objective>
Write the fping shadow capture script and its unit tests. The script runs standalone
on cake-shaper, importing FpingMeasurement/FpingThread directly from /opt/wanctl/src,
reading configs/spectrum.yaml for reflectors and source IP, and logging:
  1. A run_start metadata record on startup (provenance: source_ip, reflectors, config path,
     script version/commit, cadence settings).
  2. Per-burst probe_cycle records for every completed burst — success OR all-loss — providing
     the raw per-burst RTT distribution AND the per-burst cycle-timing distribution that Phase 248
     needs for two DISTINCT full-window p99 values (RTT p99 from rtt_ms; cycle-timing p99 from
     elapsed_ms).
  3. Inferred dropped-burst records (when fping returned all-loss and FpingThread cached nothing)
     with elapsed_ms=null and inferred=true — honest about the fact that the real burst timing
     is not observable through FpingThread.get_latest().
  4. Periodic probe_stats summary snapshots (every STATS_INTERVAL_PROBES bursts) as a
     convenience summary layer; these are NOT the authoritative p99 source.

Purpose: PROF-01 — run fping in shadow alongside the live icmplib backend without touching
any controller-path file. The NDJSON output is the evidence input for Phase 248 distribution
analysis. PROF-01 explicitly requires BOTH raw RTT samples AND cycle p99 timing, so the
script must preserve both rtt_ms and elapsed_ms per burst.

The core correctness requirements driving this plan:
  (HIGH-fix-1) FpingThread.get_latest() returns only the cached last SUCCESSFUL sample.
  FpingMeasurement.probe() returns None on all-loss and is NOT cached, so the actual execution
  time of a failed/dropped burst is UNKNOWN via this interface. Inferred dropped records
  (detected via wall-time staleness) MUST therefore set elapsed_ms=null and inferred=true.
  Fabricating an elapsed value for a burst we never observed is forbidden — it would poison the
  cycle-timing p99.
  (HIGH-fix-2) PROF-01 requires cycle p99 TIMING, not only RTT p99. probe_cycle.rtt_ms gives
  RTT p99; probe_cycle.elapsed_ms gives the cycle-timing p99 (probe_elapsed_p99_ms_full_window).
  These are two distinct first-class outputs. The timing p99 is computed from SUCCESSFUL records
  only (inferred/dropped records have elapsed_ms=null and are excluded).
  (HIGH-fix-3) Per-burst probe_cycle records are the authoritative evidence: the OperationProfiler
  has max_samples=1200 (~3.3h at 10s cadence), so a 12h soak's hours 0-8.7 are NOT in the
  final rolling-window stats. probe_cycle records written per burst are always available for
  the full soak duration. Phase 248 MUST compute both p99 values from probe_cycle records, not
  from probe_stats snapshots.
  (MEDIUM-1) Failed/all-loss cycles must be logged as first-class probe_cycle records so
  failure periods do not bias the p99 interpretation.
  (MEDIUM-2) The probe_cycle record must include per_host_results, per_host_loss, active_hosts,
  and successful_hosts (already present in RttSample) so Phase 248 can diagnose reflector-specific
  skew or partial loss.
  (MEDIUM-3) A run_start metadata record makes evidence provenance self-contained in the
  NDJSON file without requiring the config file to reconstruct what ran.
  (MEDIUM-4) cfg['ping_source_ip'] MUST be mapped to the 'source_ip' constructor key
  FpingMeasurement expects (wrong key → wrong source interface → invalid evidence).
