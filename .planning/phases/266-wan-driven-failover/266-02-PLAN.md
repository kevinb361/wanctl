# Wave 2: Daemon Integration

**Phase**: 266
**Wave**: 2
**Scope**: Wire FailoverBridge into daemon cycle + config schema

## Tasks

1. Add config parsing for `route_management.failover` section:
   - `SteeringConfig` gains `failover_enabled`, `failover_wan`, `failover_red_cycles`, `failover_green_cycles`
   - Defaults: enabled=false, red_cycles=3, green_cycles=5
   - Schema validation in config validators

2. Wire FailoverBridge into daemon cycle:
   - Initialize bridge in `__init__` (after route_manager creation)
   - In the main cycle, after congestion assessment and before abort checks:
     - If failover enabled → call `bridge.update(congestion_state)`
     - If bridge returns action → call `route_manager.plan_or_apply(action, "spectrum")`
   - Log every failover/recovery decision with evidence

3. Add `last_failover_event` to route_manager status_snapshot (health endpoint):
   - Tracks: action (disable/enable), timestamp, consecutive_cycles, congestion_state
   - Visible in `:9102` health endpoint under route_management section

4. Config reload: re-initialize FailoverBridge on SIGUSR1 if failover config changed

## Deliverables
- Updated `src/wanctl/steering/daemon.py` (bridge initialization + cycle integration)
- Updated `src/wanctl/steering/config.py` (failover config fields)
- Updated `src/wanctl/steering/route_manager.py` (status_snapshot addition)
- Updated config validators
- Steering config schema tests

## Constraints
- Failover bridge is a passive observer — does not modify congestion detection
- Only Spectrum WAN is congestion-reactive (ATT static)
- Abort path (circuit breaker, Netwatch contention) still overrides failover
