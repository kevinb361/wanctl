# Plan 249-01 — Autorate Flat-Gauge Fire-on-Change Audit

## Objective

Audit per-metric write rates on both WAN metrics databases and apply the steering fire-on-change pattern only to confirmed current flat gauges emitting at >=2Hz with near-zero value variance.

This phase is intentionally audit-first. If the current stable live deployment has no confirmed candidates, the correct outcome is no code change.

## Constraints

- Read-only live DB inspection is allowed.
- No service restarts, config changes, RouterOS changes, or production ownership changes.
- Do not optimize metrics observed only during prior native wanctl canaries unless they are present in the current stable deployment window.
- Storage hygiene must not introduce unrelated controller-path behavior drift.

## Live audit method

Use `wanctl-history --ingestion-rate --by-table --rolling=60,300,3600 --json` semantics against both per-WAN DBs.

Operational note from live inspection: on `cake-shaper`, `/usr/local/bin/wanctl-history` is a stale symlink to `/opt/wanctl/scripts/wanctl-history`, which is absent. The module path works with sudo as:

```bash
sudo -n env PYTHONPATH=/opt python3 -m wanctl.history \
  --ingestion-rate --by-table --rolling=60,300,3600 --json
```

Because normal discovery touches `/var/lib/wanctl/metrics.db`, sudo is needed for read-only DB discovery.

## Candidate selection rule

A metric is a Phase 249 implementation candidate only if all are true:

1. It appears in the current stable 60s or 300s window at >=2Hz.
2. It has near-zero variance in the same stable window.
3. Consumers can tolerate sparse last-value-style semantics.
4. It is not a transition/event metric where per-sample history is semantically meaningful.

The 3600s window is useful for historical context but not sufficient by itself when it includes earlier native canaries or other temporary mode changes.

## Expected implementation if candidates exist

For each confirmed candidate:

1. Add a per-instance last-emitted cache initialized to `None`.
2. Always emit the first value after process start.
3. Emit again only when the value changes.
4. Keep labels stable and include labels in cache identity when labels affect metric semantics.
5. Add unit tests using the `SimpleNamespace`/direct-method pattern from `tests/steering/test_steering_metrics_recording.py::TestSteeringEnabledFireOnChange`.
6. Run a short read-only before/after ingestion-rate measurement; if a service restart would be required to activate the change, keep it explicitly gated and reversible.

## Verification gates

Read-only/audit no-op gates:

- 60s and 300s windows show zero confirmed >=2Hz flat-gauge candidates on both WANs.
- Any 3600s candidates are classified as contaminated by earlier canaries or non-current mode and are not mutated in this phase.
- GAUGE-02 and GAUGE-03 close as vacuous/empty-set complete: every confirmed candidate was changed/tested, because no current candidates were confirmed.
- No source files change.

Implementation gates, if candidates exist:

- Candidate-specific RED/GREEN tests.
- Relevant focused tests for autorate/steering metrics recording.
- `ruff check` on touched files.
- `mypy` on touched source files.
- `git diff --check`.
- Before/after ingestion-rate evidence.
