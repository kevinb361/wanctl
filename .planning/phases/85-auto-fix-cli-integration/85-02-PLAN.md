    - Test: run_fix() returns user-cancelled result when confirmation declined
    - Test: _apply_changes() returns CheckResult PASS per param on success, ERROR on failure
    - Test: _apply_changes() category is "Fix Applied (download)" or "Fix Applied (upload)"
  </behavior>
  <action>
Add to check_cake.py in the `# FIX INFRASTRUCTURE` section (after the functions from Plan 01):

**1. _show_diff_table(changes_by_direction: dict[str, dict[str, tuple[str, str]]], queue_names: dict[str, str]) -> int:**
- Print table per direction: "Proposed changes for {queue_name} ({direction}):"
- Columns: Parameter | Current | Recommended (aligned with padding)
- Strip "cake-" prefix from param names for display
- Return total change count
- Print to stderr (not stdout, so --json output stays clean)

**2. _confirm_apply(total_changes: int) -> bool:**
- Prompt: f"Apply {total_changes} changes? [y/N] " via input()
- Return True only for "y", "Y", "yes", "YES"
- Default (empty/anything else) = False (safe default)

**3. _apply_changes(client: object, changes_by_direction: dict[str, dict[str, tuple[str, str]]], queue_names: dict[str, str]) -> list[CheckResult]:**
- For each direction with changes: build params dict `{key: recommended_value}` and call `client.set_queue_type_params(queue_name, params)`
- Single PATCH per queue type (all changed params in one call) -- RouterOS PATCH is atomic per resource
- On success: one CheckResult PASS per changed param, category "Fix Applied ({direction})"
- On failure: one CheckResult ERROR per param in that direction, category "Fix Applied ({direction})"

**4. run_fix(data: dict, config_type: str, client: object, yes: bool = False, json_mode: bool = False, wan_name: str = "") -> list[CheckResult]:**
Orchestration flow:
1. check_daemon_lock() -- if ERROR, return immediately
2. Run audit via existing step 3.5 logic (get queue type data per direction, run _extract_changes_for_direction)
3. If no changes: return PASS result "All CAKE parameters are optimal -- nothing to fix."
4. If json_mode and not yes: return ERROR "Fix in --json mode requires --yes flag"
5. If not yes: _show_diff_table() + _confirm_apply() -- if declined, return "Fix cancelled by user" PASS
6. _save_snapshot(queue_type_data_by_direction, wan_name) -- print snapshot path to stderr
7. _apply_changes(client, changes, queue_names) -- collect results
8. Re-run full run_audit(data, config_type, client) for verification
9. Return all results (lock PASS + apply results + verification results)

The function needs to fetch queue type data itself (like step 3.5 in run_audit does) to get both the data for change extraction and the queue type names for PATCH targeting.

Tests in new classes: TestFixFlow, TestShowDiffTable, TestConfirmApply, TestApplyChanges. Mock client.set_queue_type_params, client.get_queue_types, client.get_queue_stats. Use capsys/capfd for table output verification. Mock input() for confirmation tests.
