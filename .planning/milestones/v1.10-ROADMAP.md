# v1.10 Architectural Review Fixes Roadmap

## Overview

v1.10 addresses findings from a senior architectural review. The milestone works through issues in priority order: critical hot-loop and transport bugs that risk production stability, steering reliability gaps, operational resilience hardening, code cleanup, codebase audit, and test quality improvements. No new features -- this is hardening only.

## Phases

- [ ] **Phase 50: Critical Hot-Loop & Transport Fixes** - Fix blocking delays in hot loop, transport config contradictions, and failover re-probe
- [ ] **Phase 51: Steering Reliability** - Fix state normalization, anomaly detection semantics, stale baseline detection, and file safety
- [ ] **Phase 52: Operational Resilience** - SSL defaults, DB corruption recovery, disk monitoring, CVE patch, config error messages
- [ ] **Phase 53: Code Cleanup** - Rename misleading variables, fix stale docstrings, clean imports, scope warnings, ruff fixes, extract function
- [ ] **Phase 54: Codebase Audit** - Audit duplication, module boundaries, and remaining complexity hotspots
- [ ] **Phase 55: Test Quality** - Consolidate fixtures, add behavioral integration tests, reduced-mock tests, failure cascade tests

## Phase Details

### Phase 50: Critical Hot-Loop & Transport Fixes

**Goal**: The autorate hot loop runs without multi-second blocking delays, transport selection honors configuration, and failover automatically recovers to primary transport
**Depends on**: Nothing (first phase, highest production risk)
**Requirements**: LOOP-01, LOOP-02, LOOP-03, LOOP-04, CLEAN-04
**Success Criteria** (what must be TRUE):

1. Retry decorator in the autorate hot loop blocks for at most 50ms per attempt with a single retry, never the previous 1s+2s delays
2. The `router_transport` config setting controls which transport is used as primary -- setting `rest` uses REST first, setting `ssh` uses SSH first, with no contradictions between config defaults and factory defaults
3. After falling back to SSH, the failover client periodically re-probes REST and transparently restores it when available
4. Main autorate loop exits within one cycle interval of receiving SIGTERM/SIGINT (uses shutdown_event.wait instead of time.sleep)

**Plans**: 3 plans

- [ ] 50-01-PLAN.md -- Sub-cycle retry delays on run_cmd + shutdown_event.wait in main loop (LOOP-01, LOOP-04)
- [ ] 50-02-PLAN.md -- Make config.router_transport authoritative for transport selection (LOOP-02, CLEAN-04)
- [ ] 50-03-PLAN.md -- Add periodic re-probe of primary transport after failover (LOOP-03)

### Phase 51: Steering Reliability

**Goal**: Steering daemon handles edge cases correctly -- legacy state names, anomaly detection semantics, stale baselines, and file read safety
**Depends on**: Phase 50 (transport fixes may affect steering's router communication)
**Requirements**: STEER-01, STEER-02, STEER-03, STEER-04
**Success Criteria** (what must be TRUE):

1. When `_is_current_state_good()` encounters a legacy state name and normalizes it, a warning is logged identifying the legacy name and its normalized form
2. Anomaly detection returns a cycle-skip (True) instead of cycle-failure (False), so anomalous readings cause the cycle to be skipped without incrementing the failure counter
3. BaselineLoader detects when the autorate state file is older than 5 minutes and logs a warning, degrading gracefully instead of using stale data silently
4. BaselineLoader reads the autorate state file through `safe_json_load_file()` instead of raw open/json.load, handling corruption and concurrent writes safely

### Phase 52: Operational Resilience

**Goal**: Production deployment handles real-world operational issues -- SSL certificate verification, database corruption, disk pressure, known CVEs, and actionable error messages
**Depends on**: Phase 50 (REST client SSL changes build on transport fixes)
**Requirements**: OPS-01, OPS-02, OPS-03, OPS-04, OPS-05
**Success Criteria** (what must be TRUE):

1. REST API client defaults to `verify_ssl=True`; documentation explains how to configure self-signed certificates and when disabling verification is appropriate
2. SQLite metrics writer runs `PRAGMA integrity_check` at startup and automatically rebuilds the database file if corruption is detected
3. Health endpoint response includes disk space status for `/var/lib/wanctl/`, surfacing a warning when free space drops below a configured threshold
4. The `cryptography` dependency version is updated to a release that patches CVE-2026-26007
5. YAML configuration parse errors include the line number where the error occurred, enabling users to find and fix config issues without guessing

### Phase 53: Code Cleanup

**Goal**: Codebase naming, documentation, imports, and structure accurately reflect current architecture -- no misleading variable names, stale docstrings, or unnecessary complexity
**Depends on**: Phase 52 (cleanup should happen after functional changes to avoid merge conflicts)
**Requirements**: CLEAN-01, CLEAN-02, CLEAN-03, CLEAN-05, CLEAN-06, CLEAN-07
**Success Criteria** (what must be TRUE):

1. `self.ssh` in autorate is renamed to `self.client` (or equivalent) reflecting that it holds a `FailoverRouterClient`, not an SSH connection
2. All docstrings referencing "2-second control loop" or similar outdated timing are updated to reflect the 50ms cycle interval
3. Hot loop uses module-level `time` import; no function-scoped `import time as time_module` aliasing remains
4. `InsecureRequestWarning` suppression is scoped to the REST API session instead of applying process-globally via `urllib3.disable_warnings`
5. All ruff violations are resolved and `validate_config_mode()` is extracted from `main()` to reduce its complexity score

**Plans**: TBD

### Phase 54: Codebase Audit

**Goal**: Systematic audit identifies and addresses remaining structural issues -- code duplication, module boundary clarity, and complexity hotspots beyond what previous milestones addressed
**Depends on**: Phase 53 (audit after cleanup to assess true remaining issues)
**Requirements**: AUDIT-01, AUDIT-02, AUDIT-03
**Success Criteria** (what must be TRUE):

1. Audit report documents remaining code duplication between autorate and steering daemons, with consolidation applied where benefit exceeds risk
2. Module boundaries are reviewed: `__init__.py` exports are intentional (no wildcard re-exports), import structure follows a clear layered pattern, and any confusing cross-module dependencies are documented or resolved
3. Complexity hotspots beyond `main()` are identified (cyclomatic complexity > 10), and the highest-risk ones are addressed through extraction or simplification

### Phase 55: Test Quality

**Goal**: Tests are maintainable, behavioral, and cover realistic failure scenarios -- reduced fixture duplication, integration coverage, and multi-failure testing
**Depends on**: Phase 54 (test changes should target the final code structure)
**Requirements**: TEST-01, TEST-02, TEST-03, TEST-04
**Success Criteria** (what must be TRUE):

1. Duplicated `mock_config` fixtures across 8+ test files are consolidated into shared `conftest.py` fixtures, with each test file using the shared version instead of defining its own
2. Behavioral integration tests verify autorate-steering daemon interaction through the state file interface (autorate writes state, steering reads it, decisions are correct)
3. Reduced-mock behavioral tests exercise the router communication layer (REST and SSH) with realistic request/response patterns instead of fully mocked internals
4. Failure cascade tests verify correct behavior when multiple failures occur simultaneously (e.g., router unreachable + storage error), confirming the system degrades gracefully without crashing

### Phase 56: Integration Gap Fixes

**Goal**: Close residual integration gaps found by milestone audit -- verify_ssl semantic contradiction and stale CONFIG_SCHEMA.md transport default
**Depends on**: Phase 55 (gap closure phase, runs after all functional work)
**Requirements**: OPS-01 (integration), CLEAN-04 (integration)
**Gap Closure**: Closes gaps from v1.10-MILESTONE-AUDIT.md
**Success Criteria** (what must be TRUE):
