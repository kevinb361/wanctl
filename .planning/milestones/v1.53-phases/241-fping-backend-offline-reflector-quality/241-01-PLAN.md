fping subprocess return-code semantics (review M1 — ONLY {0,1,2} are parseable):
  0  = all hosts reachable; 1 = some/all unreachable (normal loss); 2 = any IP unresolved.
  These three (and ONLY these three) are PARSEABLE loss outcomes -> parse the combined stream.
  A returncode > 2 (usage/config error, e.g. 3/4) is NOT a loss outcome -> treat as subprocess/config
  FAILURE -> _log_failure -> return None; do NOT parse it and do NOT feed the scorer.
  A NEGATIVE returncode (signal/process death, e.g. -15 SIGTERM, -9 SIGKILL) is process
  death -> treat as failure -> _log_failure -> return None; do NOT parse its partial output
  as normal loss. Combine with the TimeoutExpired/empty-output guards.
  CROSS-REF Plan 03: the captured process_death fixture's recorded returncode MUST be the
  Python-equivalent NEGATIVE value (e.g. -15), so this negative-returncode gate is exercised
  against real captured metadata (Plan 03 acceptance criterion).
