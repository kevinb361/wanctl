# v1.11 WAN-Aware Steering Roadmap

## Overview

v1.11 closes the gap where steering's failover decisions rely solely on local CAKE queue stats, which can mask ISP-level congestion. The milestone pipes autorate's end-to-end WAN RTT state (GREEN/YELLOW/SOFT_RED/RED) into steering's confidence model. SOFT_RED (CAKE exhausted at floor) and RED can independently trigger steering after sustained confirmation — the key insight is that SOFT_RED means local shaping is exhausted, so the only remaining lever is routing to ATT. Total scope is approximately 60 lines of production code across 5 existing files, plus ~150-200 lines of tests.

## Phases

- [ ] **Phase 56: WAN State Data Path** - Autorate persists congestion state; steering reads it with staleness and backward-compatibility guards
- [ ] **Phase 57: Signal Fusion** - WAN state feeds into confidence scoring as secondary weighted signal with safety constraints
- [ ] **Phase 58: Observability & Configuration** - Health endpoint, structured logging, and YAML configurability for WAN state integration

## Phase Details

### Phase 56: WAN State Data Path

**Goal**: Autorate's download congestion zone is available to steering through the shared state file, with staleness detection and backward compatibility
**Depends on**: Nothing (first phase -- adds data that existing consumers silently skip)
**Requirements**: INFRA-01, INFRA-02, INFRA-03, INFRA-04
**Success Criteria** (what must be TRUE):

1. Autorate writes its current download congestion zone (GREEN/YELLOW/SOFT_RED/RED) to the state file every cycle, verifiable by inspecting the JSON on disk
2. Steering reads the WAN congestion zone from the state file without performing a second file read (reuses the existing state file load in BaselineLoader)
3. When the WAN state timestamp is older than 5 seconds, steering treats it as unknown and it contributes zero weight to confidence scoring
4. When the congestion key is missing from the state file (pre-v1.11 autorate or fresh deploy), steering proceeds normally with no errors or warnings

### Phase 57: Signal Fusion

**Goal**: Steering's confidence model incorporates WAN congestion state — SOFT_RED/RED can independently trigger failover (CAKE exhausted at floor), YELLOW is amplifier only
**Depends on**: Phase 56 (WAN state must be available in state file)
**Requirements**: FUSE-01, FUSE-02, FUSE-03, FUSE-04, OBS-04
**Success Criteria** (what must be TRUE):

1. WAN congestion zones map to additive confidence weights: RED=70, SOFT_RED=60 (CAKE exhausted at floor — tipping point), YELLOW=5, GREEN=0
2. WAN SOFT_RED or RED can independently trigger steering after sustained 2s confirmation (existing TimerManager sustain). WAN YELLOW alone cannot trigger.
3. Sustained WAN SOFT_RED/RED steers new latency-sensitive connections to ATT via existing mangle rule mechanism
4. Recovery from failover requires WAN state to be GREEN in addition to existing CAKE GREEN requirements — WAN YELLOW or worse blocks recovery even if CAKE looks healthy
   **Plans**: TBD

### Phase 58: Observability & Configuration

**Goal**: The WAN state integration is observable through health endpoints and logs, and tunable through YAML configuration without code changes
**Depends on**: Phase 57 (WAN state must be wired into confidence model)
**Requirements**: OBS-01, OBS-02, OBS-03
**Success Criteria** (what must be TRUE):

1. Steering health endpoint (port 9102) includes WAN state details: current zone, state age in seconds, whether the state is stale, and the weight being contributed to the confidence score
2. WAN state weights (per-zone values) and staleness threshold are configurable in steering YAML, with sensible defaults matching the hardcoded values from Phase 57
3. When WAN state contributes a non-zero weight to the confidence score, the structured log line for that cycle includes the WAN zone and its weight contribution
