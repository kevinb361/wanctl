# Plan 248.2-01 — fping Freshness / Staleness Repair

## Objective

Repair the Phase 248.1 native fping canary blocker where the 50ms controller loop treated the 10s background fping producer's cached RTT as stale before the next expected fping sample, causing repeated blocking fallback checks, cycle overruns, and cycle-budget warnings.

This phase is intentionally a narrow controller-path exception to the original v1.54 SAFE-18 shadow-only invariant. The operator chose to prioritize the fping freshness fix before storage hygiene. Scope is limited to fping snapshot freshness semantics and regression tests.

## Root cause

Phase 248.1 proved that fping itself could publish healthy RTT samples, but `WANController.measure_rtt()` used fixed D-04 staleness cutoffs:

- soft warning at 0.5s
- hard failure at 5.0s

The fping background producer deliberately runs on `measurement.fping.cadence_sec` (default 10s), so a healthy cached sample can be older than 5s while still inside the expected producer cadence window. That mismatch made the loop reject the cached fping sample and run fallback checks during normal inter-sample gaps.

## Scope

Allowed changes:

1. Make `WANController.measure_rtt()` derive soft/hard stale limits from the active background producer cadence.
2. Preserve the existing fixed 0.5s/5s floors for normal icmplib behavior.
3. Add regression tests proving default 10s fping cadence accepts expected inter-sample gaps and still rejects truly stale samples.
4. Deploy the patched code to `cake-shaper` for a bounded canary.
5. Run a short native `wanctl@spectrum` + fping canary with rollback armed.
6. Roll back Spectrum ownership/config to external cake-autorate + icmplib after the canary.

Out of scope:

- Permanent production keep decision.
- Native Spectrum rate/qdisc alignment.
- Startup first-sample fallback suppression.
- Storage hygiene phases 249/250.
- RouterOS/MikroTik changes.

## Implementation

Use cadence-aware staleness limits:

- `soft_stale_sec = max(0.5, cadence_sec)`
- `hard_stale_sec = max(5.0, cadence_sec * 1.5)`

This keeps the default icmplib thread unchanged in practice because its cadence is below the existing floors, while allowing fping's 10s producer cadence to bridge expected gaps.
