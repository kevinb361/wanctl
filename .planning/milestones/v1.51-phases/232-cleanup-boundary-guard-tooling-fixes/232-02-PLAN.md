    1. Build the remote script with a fail-fast preamble: replace
       `rollback_commands_for_wan "$WAN" >"$remote_script"` with a brace group that first
       `printf '%s\n' 'set -euo pipefail'` then `rollback_commands_for_wan "$WAN"`, redirected to
       `"$remote_script"`.
    2. Fix the payload transport (cross-AI review HIGH): change the execution line to
       `ssh "${SSH_OPTS[@]}" "$SSH_HOST" "bash -s" <"$remote_script"` — i.e. drop `-n` for the
       `bash -s` payload execution ONLY. Rationale: OpenSSH `-n` redirects stdin from `/dev/null`,
       so with `-n` the real remote bash receives an EMPTY script while a stdin-reading test shim
       still sees the payload — the fix would be unproven in production. Keep `-n` on every
       read-only probe ssh call (is-active checks, preflight probes); do not refactor transport
       flags beyond this one line.
    3. Immediately after the remote script execution, add the external-writer verification from the
       review: capture `external_active="$(ssh -n "${SSH_OPTS[@]}" "$SSH_HOST" "systemctl is-active cake-autorate-${WAN}.service || true")"`;
       if it equals `active` OR `activating` (a restarting rate-control writer is still a dual-writer
       hazard), print `ROLLBACK VERIFY FAILED: cake-autorate-${WAN}.service is still ${external_active}`
       to stderr and exit 1 — BEFORE the existing wanctl@ is-active check and health check.
    4. Do not change: dry-run default mode selection, the `OPERATOR_APPROVAL` refusal gate, preflight,
       usage text, proof-JSON shape, or any command sequence emitted by `rollback_commands_for_wan` /
       `return_commands_for_wan`.

    Note on ATT sequence compatibility with `set -e`: the ATT rollback sequence already guards its
    only intentionally-fallible line (`reset_bypass_wd ... || true`), and `sleep`/`cd`/systemctl/tc
    lines are required-success steps — fail-fast is the desired semantics per the review. Do not add
    `|| true` to any other rollback line.
  </action>
