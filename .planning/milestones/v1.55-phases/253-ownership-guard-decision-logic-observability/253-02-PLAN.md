# Plan 253-02 — Multi-Signal Route Decision Logic + Startup Reconciliation/Circuit Breaker Tests

<objective>
Add route decision and active-apply safety logic that requires multi-signal WAN health, consecutive/hysteretic thresholds, startup route-state reconciliation, and circuit-breaker readiness before any active route mutation can be attempted.
</objective>

<must_haves>
<truths>
- Route failover/failback decisions must use multi-signal health and consecutive thresholds; no single-target one-sample replacement for Netwatch ships.
- Active route mutation must not run until startup reconciliation has successfully read configured route state.
- Router/API failure must open or maintain a circuit breaker and must not leave wanctl believing a route state changed.
- Phase 253 remains mock-only and does not perform live RouterOS route mutation.
- RouteManager default behavior remains safe/off or blocked unless guard/reconciliation/circuit prerequisites pass.
</truths>
<outputs>
- A route decision policy returns intended actions and evidence without mutating RouterOS.
- RouteManager or a companion model tracks reconciliation, circuit-breaker, last intended action, and last applied action.
- Unit tests prove hysteresis, reconciliation, circuit breaker, active apply gate, noop/idempotence, and failure behavior.
</outputs>
</must_haves>

<threat_model>
T-253-06 one-sample-route-flap: route failover/failback triggers from a single bad RTT/sample. Mitigation: decision policy requires consecutive unhealthy/healthy samples and multiple evidence signals.

T-253-07 mutation-before-reconciliation: daemon starts, assumes route state, and applies a surprising route change. Mitigation: active apply blocked until configured routes are read and reconciliation status is `ok`.

T-253-08 false-applied-on-router-failure: route apply command fails but internal state records success. Mitigation: RouteManager updates `last_applied_action` only after successful command result/verification; failures open circuit and return `success=False`.

T-253-09 circuit-breaker-bypass: repeated API failures continue to issue route commands every cycle. Mitigation: circuit breaker blocks active apply after configured failure threshold until explicit reset/reconciliation.

T-253-10 SAFE-19-live-mutation: tests or implementation touch live RouterOS. Mitigation: fake router clients only; no production configs/services/router state changed.
</threat_model>

<action>
Use RED-GREEN-REFACTOR.

Create `tests/test_route_decision_policy.py` and a new module `src/wanctl/steering/route_decision.py`.

Add tests for a decision policy with concrete behavior:

1. RED congestion with supporting CAKE drops/queue and below-threshold consecutive count returns `hold` / no route action.
2. RED congestion with supporting CAKE drops/queue for `degrade_samples_required` consecutive samples returns an intended failover/prefer-alternate action.
3. YELLOW or elevated RTT without required supporting signals does not produce route mutation intent.
4. GREEN recovery must be sustained for `recover_samples_required` consecutive samples before returning a failback/prefer-primary action.
5. Router unreachable / route circuit open / stale or missing route-state input returns `hold` with reason explaining the blocked precondition.
6. The result includes an evidence dict containing at least `congestion_state`, `rtt_delta_ms`, `cake_drops`, `queued_packets`, `degrade_count`, and `recover_count`.

Implement a small policy model, for example:

- `RouteDecisionSignals(congestion_state: str, rtt_delta_ms: float, cake_drops: int, queued_packets: int, router_reachable: bool, route_state_known: bool, circuit_open: bool)`
- `RouteDecisionState(degrade_count: int = 0, recover_count: int = 0, current_preference: str = "primary")`
- `RouteDecisionResult(decision: Literal["hold", "prefer_primary", "prefer_alternate"], route_actions: tuple[...], reason: str, evidence: dict[str, object], state: RouteDecisionState)`
- `RouteDecisionPolicy.evaluate(signals, state) -> RouteDecisionResult`

Do not call `RouteManager.plan_or_apply()` from the policy. The policy is pure decision logic; mutation stays in RouteManager / daemon integration.
</action>

Extend `tests/test_route_manager.py` for startup reconciliation and circuit-breaker behavior:

1. `RouteManager.reconcile_startup()` reads all configured route anchors using fake router `run_cmd('/ip route print detail where ...', capture=True)` or a helper with equivalent command strings.
2. Reconciliation succeeds only when every configured route resolves to exactly one route with known disabled/enabled state.
3. Reconciliation failure for missing route, ambiguous route, unparseable JSON, or non-zero router command result sets reconciliation status to failed and blocks active apply.
4. Active `plan_or_apply("disable", "spectrum")` with guard allowed but reconciliation missing returns failure, `mutated=False`, and makes no route enable/disable command call.
5. Router command failure during active apply increments a circuit failure counter / opens circuit and does not update `last_applied_action` as success.
6. Once circuit is open, further active apply attempts return failure without route enable/disable calls until a documented reset/reconcile method is invoked.
7. Successful active apply records `last_intended_action`, `last_applied_action`, route key, anchor, and timestamp-like value or monotonic placeholder.

Implement minimal state types in `route_manager.py` or a companion file, for example:

- `RouteReconciliationStatus(status: "unknown"|"ok"|"failed", routes: dict[str, RouteState], error: str | None)`
- `RouteCircuitBreaker(open: bool, failure_count: int, last_error: str | None)`
- fields/properties for `last_intended_action` and `last_applied_action`.

Keep command generation comment/id anchored and use the Phase 252 REST boundary forms. Do not add live hostnames or SSH wrappers.
</action>
