For the corruption test: write invalid JSON to the temp file directly, then verify BaselineLoader returns None without raising.

IMPORTANT: These are REAL file operations through the actual state_utils.py code path (atomic_write_json, safe_json_load_file). The point is to verify the contract between the two daemons without mocking the shared infrastructure.

  <action>
Create `tests/test_failure_cascade.py` with tests verifying correct behavior when multiple failures occur simultaneously.

Key design:
- Create WANController instances with mock config (use the shared mock_autorate_config from conftest if Plan 01 has run, otherwise define locally -- keep both working).
- Stack multiple failure injections using `patch` and `side_effect`:
  - Router: `mock_router.run_cmd.side_effect = ConnectionError("unreachable")`
  - Storage: `patch("wanctl.storage.writer.MetricsWriter.write_metric", side_effect=sqlite3.OperationalError("disk full"))`
  - State file: `patch("wanctl.state_utils.atomic_write_json", side_effect=OSError("disk full"))`
  - ICMP: `patch("wanctl.rtt_measurement.measure_rtt", return_value=None)` or mock ping to fail

For each test:
1. Set up the WANController or SteeringDaemon with appropriate mocks
2. Inject 2-3 simultaneous failures
3. Call `run_cycle()` or equivalent
4. Assert: no exception raised, function returns (may return error indicator)
5. Assert: error was logged (check logger.error or logger.warning calls)

IMPORTANT:
- The goal is to prove the system does NOT crash, not to test specific error handling (those are already covered in test_autorate_error_recovery.py). This tests the COMBINATION.
- Each test should have a clear docstring explaining which failures are stacked.
- Tests for steering daemon (Test 6) should create a SteeringDaemon with the standard steering mock_config pattern.
- Use pytest.mark.parametrize where appropriate to test different failure combinations compactly.
