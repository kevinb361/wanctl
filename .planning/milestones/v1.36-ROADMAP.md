# Roadmap: wanctl v1.36 Storage Retention And DB Footprint

## Overview

Explain the live production metrics footprint, close the shared-vs-per-WAN DB ambiguity, preserve operator-safe history verification, and materially reduce the active per-WAN DB sizes without changing controller logic.

## Phases

### Phase 177: Live Storage Footprint Investigation

**Goal**: Explain what is consuming space in production and which DB files are actually part of the runtime path  
**Depends on**: Phase 176  
**Plans**: 3 plans

Plans:

- [x] 177-01: Runtime DB-path closure and production file-role inventory
- [x] 177-02: Live DB composition and retained-history evidence capture
- [x] 177-03: Findings synthesis, operator re-check path, and Phase 178 recommendation

**Details:**
Active production DBs were identified, the retained-history window was characterized, and the dominant contributors to the 5+ GB footprint were traced to live retained content rather than runaway WAL or free-page slack.

### Phase 178: Retention Tightening And Legacy DB Cleanup

**Goal**: Reduce the active metrics footprint safely and close out any unused legacy DB path  
**Depends on**: Phase 177  
**Plans**: 3 plans

Plans:

- [x] 178-01: Make the shared `metrics.db` role explicit and close stale zero-byte file ambiguity
- [x] 178-02: Tighten per-WAN retention conservatively while preserving tuning-safe history
- [x] 178-03: Align `/metrics/history` and operator verification paths with the updated storage layout

**Details:**
Steering’s shared DB role was declared explicitly, per-WAN raw retention was tightened to 1 hour while preserving aggregate history and maintenance bounds, and operator docs plus history readers were aligned to the authoritative per-WAN topology.

### Phase 179: Verification And Operator Evidence

**Goal**: Verify in production that the new storage footprint holds and that operators can re-check it without guesswork  
**Depends on**: Phase 178  
**Plans**: 3 plans

Plans:

- [x] 179-01: Capture live post-change DB sizes and storage status against the April 13 baseline
- [x] 179-02: Prove live reader topology and retained-history shape through supported operator surfaces
- [x] 179-03: Synthesize operator evidence, close `OPER-04`, and align final docs

**Details:**
Production evidence clarified that storage health was good but the per-WAN DB footprint had not materially shrunk yet. It also closed `OPER-04` with a repeatable operator proof path for DB inventory, storage status, and history-reader validation.

### Phase 180: Verification Backfill And Audit Closure

**Goal**: Close the missing milestone verification trail for Phase 177 and leave `STOR-04` audit-safe  
**Depends on**: Phase 179  
**Plans**: 2 plans

Plans:

- [x] 180-01: Backfill the missing Phase 177 verification artifact for `STOR-04`
- [x] 180-02: Re-anchor the `STOR-04` audit trail and route the milestone back to re-audit

**Details:**
The missing Phase 177 verification artifact was created from existing evidence, removing the earlier milestone orphan and restoring a complete verification trail for the storage-investigation requirement.

### Phase 181: Production Footprint Reduction And Reader Parity

**Goal**: Close the failed production outcome behind `STOR-06` and reconcile the remaining live reader-path drift  
**Depends on**: Phase 180  
**Plans**: 3 plans

Plans:

- [x] 181-01: Implement a concrete storage-only reduction path that can materially shrink live per-WAN DB files
- [x] 181-02: Close or explicitly narrow live CLI/HTTP history-reader parity drift without breaking the HTTP contract
- [x] 181-03: Capture final production footprint and reader-parity evidence against the fixed 2026-04-13 baseline

**Details:**
This phase introduced the explicit offline compaction helper, repaired the startup/watchdog regression caused by pre-health storage work, and narrowed the reader-role contract to CLI merged reads plus endpoint-local HTTP history. Spectrum shrank materially, but ATT remained effectively unchanged, so the milestone still required a final gap-closure phase.

### Phase 182: ATT Footprint Closure

**Goal**: Deliver a materially smaller ATT per-WAN DB footprint in production and close the last open milestone requirement  
**Depends on**: Phase 181  
**Plans**: 3 plans

Plans:

- [x] 182-01: ATT reduction precheck and exact operator procedure
- [x] 182-02: ATT-only reduction execution and minimal helper alignment
- [x] 182-03: Final production proof, verification, and audit handoff

**Details:**
Phase 182 proved ATT was already mostly reclaimable free-list space, reran the ATT-only compaction path, and reduced `metrics-att.db` from about `5.08 GB` to about `202 MB`. With Spectrum already smaller, this closed `STOR-06` in production terms.

## Milestone Summary

**Status:** SHIPPED 2026-04-14  
**Phase Range:** 177-182  
**Total Plans:** 17  
**Total Tasks:** 33

**Key Accomplishments:**

- Identified the true active production DB topology and explained the multi-GB footprint with production evidence.
- Tightened per-WAN retention without changing controller logic or breaking operator-facing contracts.
- Closed `OPER-04` with a repeatable operator proof path for DB inventory, storage status, and merged CLI history verification.
- Repaired the startup/watchdog regression introduced during footprint-reduction work by bounding pre-health storage work.
- Added an explicit offline compaction path and used it to reduce ATT from about `5.08 GB` to about `202 MB`, closing `STOR-06`.

**Issues Resolved:**

- The active DB topology is now explicit: per-WAN autorate DBs plus shared steering `metrics.db`.
- The remaining v1.36 footprint blocker was reduced to an ATT-only compaction outcome and then closed.
- Startup stability under the repo-default watchdog budget was restored after the storage-work regression.

**Issues Deferred:**

- The dashboard history consumer still does not surface the narrowed CLI-vs-endpoint-local HTTP distinction or `metadata.source`.

**Technical Debt Incurred:**

- Phase 181 introduced a narrowed reader-role contract that is correct and documented, but the dashboard consumer still lags behind that contract.
