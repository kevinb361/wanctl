# WireGuard TX Error Diagnosis Results

**Diagnosed:** 2026-04-05T04:24:00Z (Task 2 of 140-01-PLAN)
**Router:** RB5009 10.10.99.1 (RouterOS 7.20.7)
**Peer:** phone (72.158.231.17:3840, cellular, NOT home WiFi)

## Root Cause: ZeroTier Binding to WireGuard Interface

**ZeroTier (zt1, port 9993) was configured with `interfaces=all`, causing it to bind to
the wireguard1 interface address (10.255.255.1).** ZeroTier periodically contacts its root
servers and peers via UDP. When using 10.255.255.1 as source address, packets were routed
out through wireguard1, which failed because the destination addresses (public ZT servers
like 96.8.165.93, 84.17.53.155, 185.152.67.145) are not in any WG peer's allowed-address list.

Each failed ZT packet = 1 tx-error on the WG interface.

## Diagnosis Evidence

### Firewall Log Capture

Temporary log rules on the output chain revealed:

```
WG-TX-DEBUG output: in:(unknown 0) out:wireguard1, proto UDP, 10.255.255.1:9993->96.8.165.93:1145, len 165
WG-TX-DEBUG output: in:(unknown 0) out:wireguard1, proto UDP, 10.255.255.1:23196->96.8.165.93:1145, len 165
WG-TX-DEBUG output: in:(unknown 0) out:wireguard1, proto UDP, 10.255.255.1:26664->96.8.165.93:1145, len 165
WG-TX-DEBUG output: in:(unknown 0) out:wireguard1, proto UDP, 10.255.255.1:9993->84.17.53.155:9993, len 165
...
```

- Source: 10.255.255.1 (WG interface address)
- Ports: 9993, 23196, 26664 (ZeroTier standard port + secondary ports)
- Destinations: ZeroTier root servers (96.8.165.93, 84.17.53.155, 185.152.67.145)
- ALL on output chain (locally generated by router, NOT forwarded traffic)
- Forward chain log rule: 0 packets (confirming local origin)

### Error Pattern Match

- Errors came in bursts of ~9 (3 ports x 3 destinations) every 10-20 seconds
- Error rate was ~40,000/day (matching lifetime rate of 43,760/day)
- Errors accumulated regardless of WG peer online/offline status
- Errors accumulated with tx-packet=0 (failed before counted as sent)

### Hypotheses Tested

| Hypothesis | Result | Evidence |
| --- | --- | --- |
| MTU/fragmentation | **RULED OUT** | 0 errors even with 1392-byte pings (max inner size) |
| MSS clamp gap | **NOT PRIMARY** | Errors occur with UDP (ZT), not TCP. MSS clamp gap still exists but doesn't cause errors |
| Orphan route (10.252.255.0/24 via .3) | **RULED OUT** | Disabling route didn't change error rate |
| ZeroTier interface binding | **ROOT CAUSE** | Restricting ZT to non-WG interfaces = 0 errors |

## Fix Applied

**Changed ZeroTier `interfaces` from `all` to `bridge1,ether1-WAN-Spectrum,ether2-WAN-ATT`**

```
PATCH /rest/zerotier/*1
{"interfaces": "bridge1,ether1-WAN-Spectrum,ether2-WAN-ATT"}
```

This prevents ZeroTier from binding to wireguard1 and using 10.255.255.1 as source address.

## Additional Findings

### Orphan Route (not causing errors but misconfigured)

Static route 10.252.255.0/24 via 10.255.255.3 exists but peer .3 doesn't exist.
Not causing tx-errors (tested by disabling -- no change), but should be cleaned up.
Left in place for user to decide -- may be intentional for a future WG peer.

### WG Address Subnet (/24 vs /30)

WG interface address is 10.255.255.1/24, creating a connected route for 254 addresses
but only 1 peer (.2) exists. Not causing errors directly (ZT was the culprit), but
a /30 would be more correct for a single-peer setup.

### MSS Clamp Gap Still Exists

The postrouting MSS clamp rule does NOT apply to inner WG tunnel TCP traffic
(it sees the outer UDP packet, not inner TCP). Not causing tx-errors, but could
cause TCP performance issues through the tunnel. A forward-chain MSS clamp for
10.255.255.0/24 traffic would fix this.
