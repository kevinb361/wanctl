
**Phase 3: Hypothesis 1 -- MTU/fragmentation (per D-02)**

Test packet sizes through the tunnel. Monitor tx-error counter during each test:
```
# ping size + 28 (IP+ICMP headers) = inner packet size
# Inner 1392 (ping 1364) -> outer 1452 (safe, 48 under 1500)
# Inner 1400 (ping 1372) -> outer 1460 (safe, 40 under 1500)
# Inner 1410 (ping 1382) -> outer 1470 (safe, 30 under 1500)
# Inner 1420 (ping 1392) -> outer 1480 (20 under 1500, tight)
```

Use REST tool/ping if available, or monitor tx-error delta while generating traffic of known sizes from phone. If errors spike at specific packet sizes, MTU/fragmentation is confirmed.

**Phase 4: Hypothesis 2 -- MSS clamp gap**

Check if TCP through WG gets MSS clamped. MSS clamp is on postrouting -- does WG encapsulate before or after mangle postrouting? If WG encapsulates in forward chain, postrouting MSS clamp sees the OUTER UDP packet, not inner TCP. This would mean inner TCP never gets MSS clamped, causing oversized inner TCP segments and tx-errors.

If MSS clamp is the issue, add a clamp rule for WG inner traffic:
```bash
curl -sk -u admin:d00kie -X PUT "https://10.10.99.1/rest/ip/firewall/mangle" \
  -H "content-type: application/json" \
  -d '{
    "chain": "forward",
    "protocol": "tcp",
    "tcp-flags": "syn",
    "dst-address": "10.255.255.0/24",
    "action": "change-mss",
    "new-mss": "clamp-to-pmtu",
    "comment": "MSS clamp for WireGuard tunnel traffic",
    "passthrough": "yes"
  }'
```

**Phase 5: Hypothesis 3 -- If MTU is the root cause, reduce WG MTU**

If diagnosis points to MTU fragmentation (not just TCP MSS):
```bash
WG_ID=$(curl -sk -u admin:d00kie "https://10.10.99.1/rest/interface/wireguard" | python3 -c "import sys,json; print(json.load(sys.stdin)[0]['.id'])")

curl -sk -u admin:d00kie -X PATCH "https://10.10.99.1/rest/interface/wireguard/$WG_ID" \
  -H "content-type: application/json" \
  -d '{"mtu": "1400"}'
```
Note: Phone WG app may also need MTU update to match.
