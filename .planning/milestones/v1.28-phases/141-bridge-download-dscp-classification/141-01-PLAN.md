Create `deploy/nftables/bridge-qos.nft` -- an nft-loadable ruleset file. The file must:

1. Flush and recreate table `bridge qos` (idempotent on reload)
2. Create a `forward` chain with `type filter hook forward priority -10; policy accept;` (priority -10 to fire before any default priority 0 chains)
3. Per D-06/D-07: Early trust rule -- `ip dscp != 0 accept` for BOTH bridges. Packets with non-zero DSCP (endpoint-set EF from VoIP phones) pass through untouched.
4. Per D-05: Connection mark restore rules -- if `ct mark` is already set (from prior classification of same flow), apply the stored DSCP immediately and accept. This avoids reclassifying every packet in an established flow.
   - `ct mark 0x00000004 ip dscp set ef accept` (Voice)
   - `ct mark 0x00000001 ip dscp set cs1 accept` (Bulk)
   - Packets with `ct mark 0` fall through to classification rules below.
5. Per D-02: Voice classification (new connections only, `ct state new`):
   - DNS responses: `udp sport 53` -- set `ip dscp ef` + `ct mark set 0x00000004`
   - DNS responses: `tcp sport 53` -- set `ip dscp ef` + `ct mark set 0x00000004`
   - SIP signaling: `udp sport 5060` -- set `ip dscp ef` + `ct mark set 0x00000004`
   - RTP media: `udp sport 16384-32767` -- set `ip dscp ef` + `ct mark set 0x00000004`
6. Per D-03: Bulk classification (new connections only):
   - BitTorrent: `tcp sport 6881-6889` -- set `ip dscp cs1` + `ct mark set 0x00000001`
   - Usenet NNTPS: `tcp sport 563` -- set `ip dscp cs1` + `ct mark set 0x00000001`
7. Per D-03: Large-flow demotion via conntrack bytes:
   - `ct bytes > 10000000` (10MB) + `ct mark 0` (not already classified) -- set `ip dscp cs1` + `ct mark set 0x00000001`
   - This catches large unclassified downloads and demotes them to Bulk.
8. Per D-04: Best Effort is the default -- no rule needed, DSCP stays 0.
9. Per D-08/D-09/D-10: ALL classification rules MUST have `iif`/`oif` guards:
   - Spectrum download: `iif ens16 oif ens17`
   - ATT download: `iif ens27 oif ens28`
   - Use two separate chains or duplicate rules with different iif/oif. Separate chains per WAN is cleaner.
10. Every rule must have `ether type ip` guard for IPv4 matching.

Structure the nft file as:
```
table bridge qos {
  chain spectrum_dl { ... }
  chain att_dl { ... }
  chain forward {
    type filter hook forward priority -10; policy accept;
    iif ens16 oif ens17 jump spectrum_dl
    iif ens27 oif ens28 jump att_dl
  }
}
```

Each per-WAN chain contains the identical classification rules (trust, ct mark restore, voice, bulk, large-flow demotion). No `return` at the end -- policy accept handles unmatched.
