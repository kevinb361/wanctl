**Step 1: Gather production resource baseline (per D-08)**

Observe current memory and task usage of running services to size limits appropriately:

```
ssh kevin@10.10.110.223 "systemctl show wanctl@spectrum.service -p MemoryCurrent,TasksCurrent 2>/dev/null"
ssh kevin@10.10.110.223 "systemctl show wanctl@att.service -p MemoryCurrent,TasksCurrent 2>/dev/null"
ssh kevin@10.10.110.223 "systemctl show steering.service -p MemoryCurrent,TasksCurrent 2>/dev/null"
```

Also check peak RSS from process stats:
```
ssh kevin@10.10.110.223 "ps -u wanctl -o pid,rss,vsz,args --no-headers"
```

**Step 2: Determine resource limit values (per D-08 -- based on observation, not arbitrary)**

Use the following sizing strategy:
- MemoryMax = 2x observed peak memory (gives headroom for metric accumulation and GC pressure)
  - Expected: wanctl processes ~100-200MB each, so MemoryMax=512M is generous
  - steering is lighter, so MemoryMax=256M
  - If observed values are different, adjust accordingly
- TasksMax = observed threads + 50% headroom
  - Expected: ~15-25 threads per wanctl process, so TasksMax=64
  - steering: ~10 threads, so TasksMax=32
- LimitNOFILE = 1024 (wanctl opens metrics.db + log files + network sockets, nowhere near default 1024 limit, but set explicitly for documentation)

**Step 3: Add resource limits to wanctl@.service**

SSH to VM, read the current (already hardened) wanctl@.service, and add resource limit directives in the [Service] section:

```ini
# Resource limits (per D-07, D-08 -- sized from production observation)
MemoryMax=512M
MemoryHigh=384M
TasksMax=64
LimitNOFILE=1024
```

Add these lines after the existing hardening directives. Use `sudo tee` or `sudo sed` to insert.

MemoryHigh=384M provides a soft limit that triggers kernel memory pressure reclaim before the hard MemoryMax kills the service.

**Step 4: Add resource limits to steering.service**

```ini
# Resource limits
MemoryMax=256M
MemoryHigh=192M
TasksMax=32
LimitNOFILE=1024
```

Steering is lighter weight -- it does not maintain CAKE queues or run tc commands.

**Step 5: Verify NIC tuning service is complete (per D-05)**

Read the current wanctl-nic-tuning.service on the VM (already updated in Plan 01 with hardening). Verify it contains ALL required ethtool commands:
- Ring buffers: `-G ensXX rx 4096 tx 4096` for all 4 NICs
- GRO forwarding: `-K ensXX rx-udp-gro-forwarding on` for all 4 NICs
- IRQ affinity: CPU pinning for Spectrum NICs (CPU 0) and ATT NICs (CPU 1)

The existing service already has all these commands. The persistence comes from the service being enabled (`systemctl is-enabled wanctl-nic-tuning.service` should return "enabled"). Verify:

```
ssh kevin@10.10.110.223 "systemctl is-enabled wanctl-nic-tuning.service"
```

If not enabled:
```

```
