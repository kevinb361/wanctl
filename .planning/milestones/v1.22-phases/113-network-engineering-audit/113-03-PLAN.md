1. Per D-14, capture CAKE statistics at IDLE (no load):
   Wait for a period with no active downloads/uploads (or verify via quick check), then capture:
   ```
   ssh kevin@10.10.110.223 "sudo tc -s qdisc show"
   ```
   Save the full output. Parse and document for each WAN (br-spectrum, br-att) and each direction:
   - Total sent (bytes, packets)
   - Drops, overlimits, requeues
   - Current backlog (bytes, packets)
   - memory_used vs memory_limit
   - Per-tin breakdown if available (Bulk/BestEffort/Video/Voice)

   Also capture JSON format for machine-readable baseline:
   ```
   ssh kevin@10.10.110.223 "sudo tc -j -s qdisc show"
   ```

2. Per D-14 and D-15, capture CAKE statistics UNDER LOAD:
   Use wanctl-benchmark or flent to generate load. The simplest approach is a single-flow TCP download test:
   ```
   ssh kevin@10.10.110.223 "sudo tc -s qdisc show" > /tmp/pre_load.txt
   ```
   Then start a brief load test (flent single TCP download for ~30 seconds):
   ```
   ssh kevin@10.10.110.223 "flent tcp_download -H dallas -l 30 -t 'NETENG-05 load test'"
   ```
   During or immediately after the load test, capture stats again:
   ```
   ssh kevin@10.10.110.223 "sudo tc -s qdisc show"
   ```

   If flent is not available on the VM, use a simpler approach:
   ```
   ssh kevin@10.10.110.223 "iperf3 -c dallas -t 30 &" && sleep 15 && ssh kevin@10.10.110.223 "sudo tc -s qdisc show"
   ```
   Or use curl to a large file to generate download load.

   IMPORTANT: Do NOT leave load running. Ensure any load test completes or is killed.

3. Document the operating range in a comparison table:

   | Metric | Idle | Under Load | Notes |
   |--------|------|------------|-------|
   | Backlog (bytes) | X | Y | Queue depth |
   | Backlog (packets) | X | Y | Packet count |
   | Drops | X | Y | Expected to increase under load |
   | memory_used | X | Y | vs memory_limit |
   | memory_used % | X% | Y% | Headroom indicator |

   Create tables for:
   - Spectrum download (br-spectrum ingress)
   - Spectrum upload (br-spectrum egress)
   - ATT download (br-att ingress)
   - ATT upload (br-att egress)

4. Analyze memory pressure:
   - If memory_used/memory_limit > 80% under load, flag as potential concern
   - If memory_used/memory_limit < 20% under load, note memlimit may be oversized
   - Document whether 32mb memlimit is appropriate for the configured bandwidth

5. Document per-tin stats if available:
   - Which tins see the most traffic (expected: Best Effort)
   - Which tins see drops (expected: Best Effort and Bulk during congestion)
   - Voice tin should show minimal drops and low backlog

Write all findings to .planning/phases/113-network-engineering-audit/113-03-findings.md under "## NETENG-05: Queue Depth and Memory Pressure Baseline".
