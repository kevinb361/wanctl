# Roadmap: wanctl v1.37 Dashboard History Source Clarity

## Overview

Make the dashboard history tab reflect the current history contract so
operators can tell when they are looking at endpoint-local HTTP history, see
the source metadata that explains it, and know how to reach the authoritative
merged CLI history path without changing backend control or storage behavior.

## Phases

### Phase 183: Dashboard History Contract Audit

**Goal**: Close the gap between the documented history contract and what the dashboard currently implies  
**Depends on**: Phase 182  
**Plans**: 2 plans

Plans:

- [x] 183-01: Audit dashboard history behavior, payload usage, and operator ambiguity points
- [x] 183-02: Define the dashboard-facing source contract and acceptance criteria for UI, tests, and docs

**Details:**
Phase 183 audited the existing dashboard history surface against the real
payload contract, documented the ambiguity between endpoint-local HTTP history
and merged CLI history, and locked a dashboard-facing source contract so later
phases could implement and verify the narrowed wording without reopening
backend `/metrics/history` semantics.

### Phase 184: Dashboard History Source Surfacing

**Goal**: Implement the dashboard changes that make local-vs-merged history semantics explicit without changing backend history behavior  
**Depends on**: Phase 183  
**Plans**: 3 plans

Plans:

- [x] 184-01: Surface source scope and endpoint-local labeling in the history tab
- [x] 184-02: Expose `metadata.source` and related source context in the history UI
- [x] 184-03: Add a clear operator path from the dashboard to the authoritative merged CLI history workflow

**Details:**
Phase 184 added the pure five-state history classifier, always-mounted framing
surfaces in the History tab, translated `metadata.source` success detail and raw
diagnostics, and a compose-only merged-CLI handoff guarded against parity
language. Backend history semantics stayed unchanged; the dashboard simply
consumed the existing source metadata more explicitly.

### Phase 185: Verification And Operator Alignment

**Goal**: Lock the new dashboard contract with regression coverage and operator-facing guidance  
**Depends on**: Phase 184  
**Plans**: 3 plans

Plans:

- [x] 185-01: Add focused dashboard/history regressions for source labeling and failure behavior
- [x] 185-02: Align docs and operator workflow guidance with the dashboard’s source semantics
- [x] 185-03: Verify the milestone closeout against the UI contract and operator workflow

**Details:**
Phase 185 added focused regression coverage for the locked contract, aligned
deployment/runbook/getting-started operator wording to the same HTTP-vs-CLI
history rule, and created a repo-side closeout artifact mapping the Phase 183
contract to concrete tests and docs. The milestone audit found no blocker-level
gaps, but retained non-blocking validation and test-depth debt.

## Milestone Summary

**Status:** SHIPPED 2026-04-14  
**Phase Range:** 183-185  
**Total Plans:** 8  
**Total Tasks:** 23

**Key Accomplishments:**

- Locked the dashboard history source contract so endpoint-local HTTP history,
  merged CLI proof, and degraded-source behavior are all explicit and stable.
- Made the History tab surface endpoint-local framing, translated source
  provenance, and an always-visible merged-CLI handoff without changing backend
  `/metrics/history` behavior.
- Added focused dashboard regressions covering success, fetch-error,
  source-missing, mode-missing, and db-paths-missing flows.
- Aligned `DEPLOYMENT.md`, `RUNBOOK.md`, and `GETTING-STARTED.md` to the same
  endpoint-local HTTP versus authoritative merged CLI rule.
- Closed all five `v1.37` requirements and produced a milestone audit with no
  blocker-level requirement, integration, or flow gaps.

**Issues Resolved:**

- The dashboard no longer implies that `/metrics/history` is the authoritative
  merged cross-WAN history reader.
- Operators now have the same history-source rule in the dashboard and in the
  primary docs workflow.
- The Phase 183 contract is backed by concrete regression and closeout evidence
  instead of handoff-only intent.

**Issues Deferred:**

- Mounted/live Textual confirmation of History-tab prominence and live runtime
  state transitions remains human verification debt from Phase 184.
- Nyquist validation artifacts are still missing for Phases 183-185.

**Technical Debt Incurred:**

- The dashboard regression suite still does not directly pin `#source-diagnostic`
  text content for success, fetch-error, and ambiguous-source states.
- There is no app-level regression explicitly proving the History tab mounts
  `#history-browser`.
