<objective>
Produce a read-only, evidence-backed trace map of where DSCP is set, preserved, or stripped along
the Spectrum download/upload path to CAKE ingress, satisfying DSCP-01. Deliver a capture script
that reads the live `bridge qos` nftables rule counters and CAKE-interface qdisc/filter state on
`cake-shaper`, and a `DSCP-TRACE.md` narrative documenting each stage (CRS hardware QoS trust →
Ruckus QoS mirroring → cake-shaper bridge `spectrum_dl` chain → CAKE egress on spec-router/spec-modem).
No external network gear is mutated; no CAKE mode change.
</objective>

<context>
The cake-shaper bridge `bridge-qos.nft` sets DSCP at `hook forward priority -10` on download
flows (`iif spec-modem oif spec-router` → chain `spectrum_dl`) BEFORE the CAKE egress qdisc on
`spec-router`. **CRITICAL (cycle-1 review, VERIFIED):** the `ip dscp set ef|af41|cs1` rules
(`bridge-qos.nft` L42, L47-49, L53-85, L96, L99-122) carry NO explicit `counter` statement, so
`nft list table bridge qos` may emit no per-rule counters. The original plan assumed counters exist
and treated their value as the cheapest survival signal — that premise is unproven. This plan now
PROBES counter availability and treats absence as `unknown`, never `negligible`. The DL DSCP
distribution histogram at enqueue (225-02) is the authoritative DL marking signal; the bridge
counters are a corroborating signal only when present as a bounded delta. CRS trust maps and Ruckus
QoS mirroring affect endpoint-set marks upstream and are documented (read-only RouterOS REST
reference at 10.10.99.1 is allowed for the narrative only, never mutated). Production CAKE mode is
`besteffort` (one tin), so per-tin CAKE counters cannot answer survival.
</context>

<title>Write the read-only DSCP trace capture script</title>
<action>
Create `scripts/phase225-dscp-trace.sh` modeled on `scripts/phase224-snapshot-a.sh` conventions:
`set -euo pipefail`; args `--output-dir <dir>` (required), `--ssh-host` (default `cake-shaper`), and
`--counter-window` (default 60, seconds for the before/after counter delta); `require_command` for
ssh, python3, sha256sum; refuse a non-empty `--output-dir`. All target-host access is read-only via
`ssh -o BatchMode=yes "$SSH_HOST" "sudo -n <read-cmd>"`. Capture, each to a named file under
`--output-dir`:
- `nft-bridge-qos-before.txt` ← `sudo -n nft list table bridge qos` (full ruleset, captured at T0).
- `nft-bridge-qos-after.txt`  ← same command captured after `--counter-window` seconds (T1).
- `tc-qdisc-spec-router.txt` ← `sudo -n tc -s qdisc show dev spec-router`.
- `tc-qdisc-spec-modem.txt` ← `sudo -n tc -s qdisc show dev spec-modem`.
- `tc-filter-spec-router.txt` ← `sudo -n tc filter show dev spec-router`; the `... root` and
  `... ingress` variants must each be guarded with `|| true` and, when the hook does not exist,
  write an explicit `# (unavailable: <hook>)` note into the file rather than aborting under `set -e`.
- `tc-filter-spec-modem.txt` ← `sudo -n tc filter show dev spec-modem` (same `|| true` guarding).
- `ip-link.txt` ← `sudo -n ip -d link show` (interface/bridge topology evidence).
**Counter-availability handling (the core cycle-1 fix):** detect whether the `ip dscp set` rules in
`nft-bridge-qos-before.txt` actually carry a `counter packets <N> bytes <M>` clause. Write
`bridge-mark-counters.txt` beginning with two header lines: `COUNTERS_AVAILABLE=true|false` and
`COUNTER_MODE=delta|snapshot|unavailable`.
  - If NO `counter` clause is present on the `ip dscp set` rules → `COUNTERS_AVAILABLE=false`,
    `COUNTER_MODE=unavailable`, and the file records that counter absence maps to
    `bridge_counter_signal=unknown` for the verdict (NEVER `negligible`). Do not fabricate values.
  - If counters ARE present → `COUNTERS_AVAILABLE=true`, `COUNTER_MODE=delta`, and emit per-rule
    before/after/delta packets+bytes for each `ip dscp set ef|af41|cs1` rule computed from the two
    snapshots over `--counter-window`. A single snapshot (window unusable) is recorded
    `COUNTER_MODE=snapshot` and likewise feeds the verdict as `unknown` (a cumulative snapshot is
    stale state, not a load measurement).
Write a `MANIFEST.md` (Captured timestamp, Source Posture = read-only, `--counter-window`, Artifacts
list, sha256 of each file) following the phase224 manifest style. End with a required-artifact
assertion loop that exits non-zero if any expected file is missing or empty (the tc-filter files are
considered present even when they contain only an `# (unavailable: ...)` note). The script MUST NOT
contain any executable nft `add`/`delete`/`flush`/`-f`, any executable `tc ... add/del/replace`, any
CAKE-mode write, or any reference to controller-path source files.
</action>
