Existing per-run reference flows in phase226-baseline-capture.sh (run loop ~lines 259-288),
all targeting the SAME REF_HOST:REF_PORT (default dallas:5201) concurrently:
- RRUL:           flent -l "$DURATION" -H "$REF_HOST" --local-bind "$LOCAL_BIND" ... rrul
- unmarked UDP:   iperf3 -c "$REF_HOST" -p "$REF_PORT" -u -b "$REF_UDP_RATE" -t ... --json
                  -> ref-udp-unmarked.NN.txt
- unmarked TCP:   iperf3 -c "$REF_HOST" -p "$REF_PORT" -t ... --json
                  -> ref-tcp-bulk-unmarked.NN.txt
- DSCP proof:     ref-dscp-proof.NN.txt (currently a static unmarked attestation)
- per-run artifacts are asserted non-empty in the for-file loop and sha256-summed at the end.

**SOURCE FACT (verified against repo, REVIEW HIGH 227-01 — Codex):**
`phase226-baseline-summary.py` today parses ONLY the per-run CAKE `tc -s qdisc` tin counters
(TinCounters) and the health NDJSON restart/transition rates. It does NOT open or parse the
iperf3 JSON artifacts at all — unmarked-UDP jitter/loss and unmarked-TCP throughput are NOT in
the summary today. AB-04's "marked-EF vs unmarked" comparison and the GATE-01 signals the 227-04
completeness gate reads must therefore be BUILT (new iperf-JSON parsing), not merely surfaced.

**iperf concurrency / port separation (REVIEW HIGH 227-01 — Codex):**
The marked-EF arm is a THIRD concurrent iperf3 flow. Pointing it at the same REF_HOST:REF_PORT
as the unmarked-UDP/TCP arms risks server contention, port collision, and stealing bandwidth from
the unmarked-UDP arm — which would break D-03's byte-for-byte matched-arm contract and the
apples-to-apples premise. DECISION: the marked-EF arm uses a DISTINCT port, default `REF_PORT+2`
(5203), exposed as `--ef-ref-port` (default = REF_PORT+2). This requires the reference reflector
(`dallas`) to have an iperf3 `--server` listening on that port; record that prerequisite in the
manifest and in the operator runbook (227-03). When --marked-ef is OFF, no new port is used and
the harness is byte-for-byte identical to 226.

The default marking mechanic (Claude's Discretion, D-05): iperf3 supports `--dscp ef`
(maps to DSCP 0x2e / TOS 0xb8). If the running iperf3 build rejects `--dscp`, fall back to
`--tos 184`. Record which path was used and whether the egress DSCP was confirmed.

**iperf-JSON validity (REVIEW MEDIUM→folded, 227-01 — Codex):** a non-empty artifact does NOT
prove a successful flow — iperf3 error text is also non-empty. Each iperf JSON (unmarked UDP,
unmarked TCP, marked EF) must be validated: parses as JSON, has no top-level `error` key, and
contains the expected `end.sum` (UDP/sender) / `end.sum_received` (TCP) block. A flow that fails
validation is recorded as invalid in the per-run proof (and, for EF, as a degraded best-effort
arm per D-05).

**Dataset routing — GATE-01 vs AB-04 (REVIEW HIGH 227-01 — Codex):**
- GATE-01 (Phase 228 accept/rollback) reads the MATCHED arms (RRUL p99, restart/transition,
  UL-stability, per-tin separation) from the EF-loaded captures — the EF arm is present in both
  the besteffort and candidate captures, so it is a constant across the A/B comparison and does
  not break the matched-load contract as long as it runs on its own port.
- AB-04 (realtime-flow protection) reads the marked-EF-vs-unmarked-UDP jitter/loss comparison
  from those same captures. The summary labels the EF-loaded captures explicitly; they are NOT
  silently reused as a hypothetical no-EF GATE-01 baseline.
