# v1.9 Performance & Efficiency Roadmap

## Overview

v1.9 reduces cycle utilization from 60-80% to ~40% at 50ms intervals through profiling-driven optimization. Profile first, optimize what matters, then add telemetry for ongoing visibility. Covers both autorate and steering daemons.

## Phases

- [ ] **Phase 47: Cycle Profiling Infrastructure** - Instrument both daemons with per-subsystem timing and collect production data
- [ ] **Phase 48: Hot Path Optimization** - Optimize subsystems identified by profiling (RTT, router, CAKE stats)
- [ ] **Phase 49: Telemetry & Monitoring** - Expose cycle budget metrics via health endpoints and structured logs

## Phase Details

### Phase 47: Cycle Profiling Infrastructure

**Goal**: Instrument both daemons with monotonic per-subsystem timing; collect and analyze production profiling data
**Depends on**: Nothing (first phase)
**Requirements**: PROF-01, PROF-02
**Success Criteria** (what must be TRUE):

1. Both autorate and steering daemon cycles emit per-subsystem timing (RTT measurement, router communication, CAKE stats read, state management, total cycle)
2. Profiling data is collected from production at 50ms interval for at least 1 hour
3. Analysis report identifies top 3 cycle time contributors with percentile breakdowns (P50, P95, P99)
4. Profiling overhead is <1ms per cycle (does not materially affect what it measures)

**Plans**: TBD

### Phase 48: Hot Path Optimization

**Goal**: Reduce cycle time based on profiling findings; target ~20ms average cycle (down from 30-40ms)
**Depends on**: Phase 47 (profiling data drives optimization choices)
**Requirements**: OPTM-01, OPTM-02, OPTM-03, OPTM-04
**Success Criteria** (what must be TRUE):

1. Average cycle time reduced to ≤20ms (from 30-40ms baseline), measured over 1 hour production run
2. P99 cycle time reduced to ≤35ms (from ~72ms baseline)
3. MikroTik RB5009 peak CPU under RRUL stress reduced from 45% (target: ≤30%)
4. No regression in congestion detection responsiveness (still sub-second)
5. All existing tests pass with no behavioral changes

**Plans**: TBD (depends on Phase 47 profiling results)

### Phase 49: Telemetry & Monitoring

**Goal**: Expose cycle budget metrics for ongoing production visibility
**Depends on**: Phase 48 (telemetry built on top of optimized paths)
**Requirements**: PROF-03, TELM-01, TELM-02
**Success Criteria** (what must be TRUE):

1. Health endpoint JSON includes cycle_time_ms (avg, P95, P99), utilization_pct, and overrun_count
2. Structured log entries include per-subsystem timing breakdowns at DEBUG level
3. Telemetry overhead is <0.5ms per cycle
4. Steering daemon health endpoint (port 9102) includes same telemetry as autorate (port 9101)
