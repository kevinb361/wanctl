<objective>
Document the root cause of the v1.56 not-ready RouterOS-inspection blocker (ACCESS-01) AND resolve
the A1 preflight (does the live RouterOS actually expose `/rest/tool/netwatch`?) as a BLOCKING gate
before any Plan 02 code is written.

The research is unambiguous on two points:
1. The failure is TWO independent broken layers, and a writeup that names only `router.key` would
   leave the real daemon-path blocker undocumented.
2. `RouteOwnershipGuard.inspect()` reads BOTH `/tool netwatch print detail` AND
   `/system script print detail` (each fail-closed). The chosen REST fix must add BOTH handlers, so
   the root-cause writeup must name BOTH missing handlers — not netwatch alone.

The A1 preflight moves EARLIER (per review HIGH-4): if the live router does not expose
`/rest/tool/netwatch`, the entire REST approach is invalid and the phase STOPS to write a separate,
properly-planned SSH fallback phase — it is NOT improvised inside Plan 03.

Purpose: ACCESS-01 requires a documented root cause covering presence, path, ownership, permissions,
and the service user for the inspection credential — AND the distinct daemon-path failure that
`router.key` repair alone would NOT fix. The A1 gate de-risks Plan 02 from being built against an
endpoint that does not exist.

Output: `258-01-root-cause.md` and `258-01-a1-preflight.md`.
</objective>
